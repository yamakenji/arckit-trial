## 2. Executive Summary

### 2.1 Overall Assessment

**Status**: ✅ APPROVED WITH CONDITIONS

The design baseline for the Order Management Modernisation project represents a well-considered, cloud-native architecture that demonstrates strong alignment with enterprise principles and requirements. The selection of Amazon EKS with Graviton3 processors, the API Gateway strangler-fig migration pattern, Aurora PostgreSQL Multi-AZ, EventBridge event bus, and AWS Step Functions saga orchestration collectively form a coherent, modern architecture suited to the project's scope.

The design scores highly on security (defence-in-depth via IRSA, WAF, KMS CMK, Security Hub, CloudTrail WORM), data integrity (transactional outbox, saga compensating transactions, Aurora Multi-AZ), and incremental risk management (strangler-fig with < 5-minute rollback per route). The Graviton3 compute choice delivers approximately 40% cost reduction against x86 equivalents, and the KEDA/Karpenter autoscaling combination provides economical elasticity for the 1,800 orders/min peak load requirement.

However, four blocking conditions prevent approval for immediate detailed design commencement. The most significant gap is the absence of any defined observability architecture — no logging pipeline, distributed tracing, metrics collection, SLI/SLO definitions, or alerting strategy appears in any of the three baseline artefacts. Additionally, the CI/CD pipeline is entirely unspecified, the DR runbook and tested RTO/RPO targets are absent, and both primary ADRs remain in unapproved statuses. These gaps must be resolved before detailed design work begins.

### 2.2 Key Strengths

- **Security depth** [ADR-001-C1, AWRS-C1]: IRSA per service account, private EKS API endpoint, WAF + VPC Link CDE boundary, Security Hub PCI DSS v4.0.1 from Day 1, CloudTrail → S3 Object Lock WORM (7 years, Compliance mode), per-customer KMS CMK for GDPR crypto-shredding — among the strongest security configurations feasible on AWS
- **Incremental migration risk management** [ADR-002-C1]: Strangler-fig with 12 discrete route migrations over 18 months, stage variable rollback in < 5 minutes, read-only routes first (Month 2) to de-risk before write paths
- **Data integrity guarantees** [ADR-001-C2, AWRS-C2]: Transactional Outbox pattern preventing dual-write failures, Step Functions Standard Workflows with compensating saga transactions, Aurora PostgreSQL Multi-AZ with sub-35s AZ failover — directly addressing P7 (Order Transaction Integrity, CRITICAL)
- **Cost-optimised compute** [ADR-001-C3, AWRS-C3]: Graviton3 m7g.2xlarge instances (~40% savings), KEDA SQS-depth scaling avoiding unnecessary pod count, Karpenter right-sizing eliminating reserved-capacity waste
- **Well-Architected alignment** [AWRS-C4]: All five classical WAF pillars rated STRONG; Sustainability rated MODERATE with improvement path identified

### 2.3 Key Concerns

- **Observability gap** [PRIN-C1]: No logging pipeline, distributed tracing, metrics collection, dashboards, SLI/SLO definitions, or alerting strategy specified in any baseline artefact — this is a BLOCKING gap against P13 (Observability) and NFR-M-003
- **CI/CD architecture absent** [PRIN-C2]: No deployment pipeline, build system, test automation, or deployment strategy (blue-green, canary) defined — violates P20 (CI/CD) and NFR-M-002
- **DR runbook and RTO/RPO validation missing** [REQ-C1]: Aurora Multi-AZ sub-35s failover is documented but no DR runbook, tested failover procedure, or validated RTO/RPO figures exist — NFR-A-003 unverified
- **Unapproved ADR statuses** [ADR-001-C4]: Both ADR-001 (PROPOSED) and ADR-002 (DRAFT) are unapproved; the design baseline lacks formal governance sign-off

### 2.4 Conditions for Approval

**MUST Address Before Detailed Design**:

1. **[BLOCKING-01]**: Define and document observability architecture — logging pipeline, distributed tracing, metrics, dashboards, SLI/SLO definitions, alerting
2. **[BLOCKING-02]**: Define CI/CD pipeline architecture — toolchain, pipeline stages, test gates, deployment strategy (blue-green or canary), environment structure
3. **[BLOCKING-03]**: Produce DR runbook with validated RTO/RPO targets and failover test plan
4. **[BLOCKING-04]**: Obtain formal approval for ADR-001 and ADR-002 before detailed design commences

**SHOULD Address During Detailed Design**:

1. **[ADVISORY-01]**: Specify resilience patterns per service — circuit breakers, bulkhead isolation, retry/backoff, timeout thresholds
2. **[ADVISORY-02]**: Define cache invalidation strategy for ElastiCache Valkey on order state change events
3. **[ADVISORY-03]**: Extend cost model with worst-case Step Functions execution volume (1,800 orders/min sustained)
4. **[ADVISORY-04]**: Define EventBridge schema registry versioning governance and backward-compatibility policy
5. **[ADVISORY-05]**: Analyse Karpenter node provisioning latency and its impact on KEDA scale-out response time under sudden demand spikes
6. **[ADVISORY-06]**: Produce formal threat model (STRIDE or equivalent)
7. **[ADVISORY-07]**: Specify SLI/SLO definitions for each of the 6 microservices

### 3.2 Principle Compliance Details

#### P1: Incremental Modernisation (CRITICAL)

**Assessment**: ✅ Compliant

**Evidence** [ADR-002-C1]:
- 12-route strangler-fig migration sequence over 18 months — textbook incremental modernisation
- Month 2 begins with read-only Order Status route, minimising initial risk
- Legacy OMS remains live throughout; rollback achievable in < 5 minutes via API Gateway stage variable update
- Month 7 completes PCI-DSS in-scope routes before tackling remaining write paths
- Month 18 completes Admin and Reporting routes last (lowest risk, highest complexity)

**Concerns**: None. This is the strongest demonstration of P1 in the design.

---

#### P2: Cloud-First Architecture

**Assessment**: ✅ Compliant

**Evidence** [AWRS-C1]:
- All compute on EKS (managed Kubernetes), no self-hosted VMs [ADR-001-C1]
- Aurora PostgreSQL (managed), ElastiCache Valkey (managed), EventBridge (serverless), SQS (serverless)
- Step Functions Standard Workflows for orchestration — fully managed, no self-hosted workflow engine
- All services confirmed available in eu-west-2 [AWRS-C5]
- No on-premises infrastructure retained in target architecture

---

#### P3: API-First Integration

**Assessment**: ✅ Compliant

**Evidence** [ADR-002-C2]:
- Amazon API Gateway Regional REST API provides unified API surface
- All 12 migration routes exposed via API Gateway, not direct service invocations
- VPC Link ensures all backend traffic routes through the API layer
- WAF applied at gateway layer, enforcing consistent API security policy

**Concerns**: OpenAPI 3.0 specification documents for each of the 6 microservices are not referenced in the baseline artefacts. This should be confirmed in DLD. [Advisory]

---

#### P4: Microservices Architecture

**Assessment**: ✅ Compliant

**Evidence** [ADR-001-C5]:
- 6 discrete microservices with clear domain boundaries: Order Submission, Inventory, Order Status, Payment, Notification, Fulfilment
- Each service runs as a separate EKS deployment with its own IRSA role (pod-level least-privilege)
- No shared-database anti-pattern; services own their data via Aurora schema separation or logical isolation
- Inter-service communication via EventBridge (async) or API Gateway (sync where required)

---

#### P5: Security by Design (NON-NEGOTIABLE/CRITICAL)

**Assessment**: ✅ Compliant

**Evidence** [AWRS-C1, ADR-001-C6]:
- IRSA (IAM Roles for Service Accounts) provides pod-level least-privilege — no node-level IAM
- Private EKS API endpoint (no public exposure of Kubernetes control plane)
- WAF + VPC Link creates auditable PCI-DSS CDE boundary at API Gateway layer
- AWS Security Hub enabled from Day 1 with PCI DSS v4.0.1, FSBP, and CIS standards
- CloudTrail → S3 Object Lock WORM (Compliance mode, 7-year retention) for immutable audit
- AWS Secrets Manager for all credentials (no hardcoded secrets)
- Per-customer KMS CMK for GDPR crypto-shredding capability
- TLS enforced on all service-to-service communication within VPC

---

#### P6: Data Sovereignty

**Assessment**: ✅ Compliant

**Evidence** [AWRS-C5, AWRS-C1]:
- All services deployed in eu-west-2 (London) — UK data residency confirmed
- S3 Object Lock for CloudTrail and data retention uses eu-west-2 buckets
- Aurora Multi-AZ within eu-west-2 (no cross-region data replication in steady state)
- DMS for migration also operates within eu-west-2

---

#### P7: Order Transaction Integrity (CRITICAL)

**Assessment**: ✅ Compliant

**Evidence** [ADR-001-C2, AWRS-C2]:
- Transactional Outbox pattern: Aurora `order_outbox` table written atomically with order state — eliminates dual-write failure between DB and EventBridge
- Outbox Relay service polls outbox table and publishes to EventBridge with exactly-once semantics
- AWS Step Functions Standard Workflows orchestrate saga with compensating transactions for multi-step order flows (payment → inventory → fulfilment)
- Aurora PostgreSQL ACID transactions ensure per-service data consistency

---

#### P8: Data Classification

**Assessment**: ⚠️ Partial

**Evidence**: Aurora stores order data including customer PII (name, address, payment reference). KMS CMK encryption is applied per-customer. [AWRS-C2]

**Gaps**: No formal data classification policy applied to individual data fields in the design. The baseline artefacts do not reference a data classification scheme (e.g., PUBLIC / INTERNAL / CONFIDENTIAL / RESTRICTED) for the entities handled by each microservice. PII fields in logs are not addressed.

**Recommendation**: Define data classification for each entity and field in DLD; ensure PII masking/redaction in structured logs (Order Submission, Payment, Notification services are highest risk).

---

#### P9: GDPR Privacy by Design

**Assessment**: ✅ Compliant

**Evidence** [AWRS-C2]:
- Per-customer KMS CMK: deleting the CMK renders all customer data cryptographically erased without destroying financial/audit records — elegant right-to-erasure implementation that satisfies Art. 17 GDPR while preserving CloudTrail WORM audit trail
- Data held in eu-west-2 (UK data residency)
- Audit trail (CloudTrail WORM) enables data access logging as required

**Minor gap**: Consent management and data subject request workflow not referenced. Confirm in DLD whether Order Submission service handles consent records or if this is out of scope. [Advisory]

---

#### P10: Event-Driven Architecture

**Assessment**: ✅ Compliant

**Evidence** [AWRS-C2, ADR-001-C7]:
- EventBridge custom bus with schema registry governs 9 domain events: OrderCreated, OrderConfirmed, PaymentProcessed, InventoryAllocated, OrderFulfilled, OrderShipped, OrderDelivered, OrderCancelled, ReturnCompleted
- SQS dead-letter queues provide durability and retry for asynchronous consumers
- Transactional Outbox ensures events are published reliably without dual-write risk
- EventBridge schema registry enforces event contract between producers and consumers

---

#### P11: Loose Coupling

**Assessment**: ✅ Compliant

**Evidence** [ADR-001-C5, AWRS-C2]:
- Services communicate via EventBridge (async publish-subscribe) — no direct service-to-service calls for domain events
- API Gateway provides synchronous integration facade; services not directly reachable from internet
- Each service owns its data; no shared schema access across service boundaries referenced

---

#### P12: Stateless Services

**Assessment**: ✅ Compliant

**Evidence** [ADR-001-C1]:
- EKS pods are stateless; all state persisted to Aurora, ElastiCache Valkey, or SQS
- Session/cache state held in ElastiCache Valkey (external, shared cache) — not in-process
- Karpenter can terminate and recreate nodes without state loss

---

#### P13: Observability

**Assessment**: ❌ Non-Compliant — **BLOCKING**

**Evidence**: None. CloudTrail is mentioned for audit, but no application-layer observability architecture is present in any baseline artefact.

**Gaps** [PRIN-C1]:
- No structured logging pipeline specified (CloudWatch Logs, log aggregation, retention policy)
- No distributed tracing solution specified (AWS X-Ray, OpenTelemetry)
- No metrics collection specified (CloudWatch Container Insights, Prometheus, or equivalent)
- No dashboards or monitoring solution specified
- No SLI/SLO definitions for any of the 6 microservices
- No alerting strategy or on-call integration specified
- No anomaly detection or SIEM integration specified

**Impact**: Without observability, incident detection and resolution will be severely hampered. This directly affects NFR-A-001 (99.9% availability), NFR-M-003, and R-001 (strategic risk) from the risk register.

**Recommendation** [BLOCKING-01]: The detailed design must include a complete observability architecture covering: structured logging, distributed tracing, metrics collection, dashboards, SLI/SLO definitions, alerting, and SIEM integration. AWS-native tooling (CloudWatch Container Insights, X-Ray, CloudWatch Dashboards) or third-party (Datadog, Grafana) should be evaluated and selected.

---

#### P14: Event-Driven Order Lifecycle

**Assessment**: ✅ Compliant

**Evidence** [AWRS-C2]:
- All 9 order lifecycle events (OrderCreated → ReturnCompleted) are enumerated and governed by EventBridge schema registry
- Event sequencing from submission through delivery and return is well-defined
- Step Functions saga manages complex multi-step flows with compensation

---

#### P15: Performance

**Assessment**: ✅ Compliant (design supports targets; validated in load test)

**Evidence** [ADR-001-C3]:
- Graviton3 m7g.2xlarge provides high compute throughput at lower latency than x86 equivalents
- ElastiCache Valkey cache-aside for Order Status reads reduces Aurora load
- Aurora 2 read replicas distribute read traffic
- KEDA scales Order Submission pods on SQS queue depth, preventing queue buildup that would inflate latency

**Caveat**: The p95 < 3s target (NFR-P-001) is supported by the design but has not been validated by load testing. A performance test plan is required before go-live. [Advisory]

---

#### P16: Scalability

**Assessment**: ✅ Compliant

**Evidence** [ADR-001-C3, AWRS-C3]:
- KEDA scales pods on SQS depth (not CPU/memory) — correctly tuned to actual workload signal
- Karpenter provisions Graviton3 nodes on demand and terminates idle nodes — no over-provisioning waste
- Node group: min 3, max 18 nodes — supports peak load of 1,800 orders/min with headroom
- Aurora read replicas scale read capacity; Aurora storage autoscales

---

#### P17: Availability & DR

**Assessment**: ⚠️ Partial

**Evidence** [ADR-001-C2, AWRS-C2]:
- Aurora PostgreSQL Multi-AZ: automatic failover < 35 seconds — strong
- EKS node group spans 3 Availability Zones with pod topology spread constraints implied
- ElastiCache Valkey cluster mode provides replication

**Gaps**:
- No DR runbook or tested failover procedure documented
- RTO/RPO targets from NFR-A-003 (< 4h RTO / < 1h RPO implied) not validated against actual recovery procedures
- Multi-region DR strategy not addressed; single-region deployment means a eu-west-2 region outage (however rare) is not mitigated
- No backup restore test schedule referenced

**Recommendation** [BLOCKING-03]: Produce a DR runbook covering: Aurora failover procedure, EKS cluster recovery, data backup restore test, and a quarterly DR drill schedule. Confirm RTO/RPO against requirements.

---

#### P18: Cost Efficiency

**Assessment**: ✅ Compliant

**Evidence** [AWRS-C3]:
- Graviton3 ~40% cost reduction vs. comparable x86
- Karpenter eliminates over-provisioning; KEDA scales to zero on idle queues
- Steady-state cost model provided: ~£9,930/month (Year 2)
- CAPEX for migration: £64,000 (ADR-002); dual-run cost: £90,000 estimated

**Minor concern**: Step Functions Standard Workflow cost at sustained 1,800 orders/min not included in worst-case cost model. [Advisory-03]

---

#### P19: IaC Everything

**Assessment**: ✅ Compliant

**Evidence** [ADR-001-C8]:
- AWS CDK TypeScript used for all infrastructure definition
- EKS cluster, node groups, IRSA roles, and Aurora provisioned via CDK constructs
- CDK code samples referenced in ADR-001

---

#### P20: CI/CD

**Assessment**: ❌ Non-Compliant — **BLOCKING**

**Evidence**: None. No CI/CD pipeline, build system, test automation stages, container image pipeline, or deployment strategy (blue-green, canary) is mentioned in any baseline artefact.

**Gaps** [PRIN-C2]:
- No CI/CD toolchain selected (GitHub Actions, AWS CodePipeline, etc.)
- No pipeline stages defined (build → unit test → integration test → security scan → deploy)
- No container image build and scan process specified
- No deployment strategy for zero-downtime updates to EKS services
- No environment structure (dev / staging / prod) defined
- No rollback automation for failed deployments

**Impact**: Without a defined CI/CD architecture, NFR-M-002 (automated deployment) cannot be met, and the project cannot claim IaC consistency (P19) at the deployment layer.

**Recommendation** [BLOCKING-02]: Define the full CI/CD architecture in detailed design. Recommend AWS CodePipeline + CodeBuild or GitHub Actions with: container image scanning (ECR + Trivy), Helm/CDK deployment to EKS, staged promotion (dev → staging → prod), and automated rollback on health-check failure.

---

#### P21: Zero-Trust Network

**Assessment**: ✅ Compliant

**Evidence** [ADR-001-C6, ADR-002-C3]:
- Private EKS API endpoint — no public Kubernetes control plane
- WAF + VPC Link at API Gateway — all traffic inspected before reaching VPC
- IRSA enforces service-account-level least-privilege; no ambient node credentials
- Security groups provide micro-segmentation within VPC
- AWS PrivateLink used for AWS service access (implied by private EKS deployment pattern)

### 10.2 Cost Concern — Step Functions at Peak Volume

At 1,800 orders/min sustained (NFR-S-001 peak), Step Functions Standard Workflow state transitions could accumulate significant cost. [ADVISORY-03]

- 1,800 orders/min × 60 min × 24h × 30 days = 77.76M orders/month (absolute peak)
- At ~5 state transitions per order = 388.8M transitions/month
- AWS Step Functions Standard: $0.025 per 1,000 state transitions = ~$9,720/month (~£7,700) at sustained peak

**Recommendation**: The cost model should include worst-case Step Functions volume. Consider Step Functions Express Workflows (pay-per-second, much cheaper for high-frequency short flows) for simpler order paths, reserving Standard Workflows for complex compensation scenarios.
