## 3. Context and Problem Statement

### 3.1 Problem Description

The Legacy Order Management System (OMS) is a monolithic application that must migrate to a cloud-native microservices architecture on Amazon EKS (per ADR-001). The organisation processes orders continuously across web, mobile, partner, and B2B channels. A planned downtime window or hard cutover is commercially unacceptable for a revenue-critical system that processes an average of 600 orders per minute and sustains peaks of 1,800 orders per minute (NFR-P-002). A decision is needed on the migration pattern and traffic routing mechanism that governs how inbound requests are progressively shifted from the legacy platform to new EKS-based microservices.

**Problem statement as a question**: What migration pattern should govern the incremental shift of order management traffic from the legacy monolith to cloud-native microservices on EKS, while maintaining zero-downtime business continuity, enabling per-route instant rollback, and enforcing PCI-DSS scope isolation by Month 7?

### 3.2 Why This Decision Is Needed

The programme must bridge from monolith to microservices without a single high-risk cutover event. This decision establishes the traffic routing mechanism, rollback strategy, PCI-DSS scope boundary, and migration sequencing framework that governs the entire 24-month programme.

- **Business context**: BR-001 mandates incremental migration with zero unplanned downtime at each phase; BR-004 requires PCI-DSS scope isolation and re-certification by Month 8 (fixed external deadline, unresolvable on the legacy platform); BR-003 requires 99.9% uptime from Month 12
- **Technical context**: NFR-A-001 (99.9% monthly uptime from Month 12), NFR-P-001 (p95 order submission < 3 seconds), NFR-P-002 (1,800 orders/min peak throughput) — the routing layer must not become a bottleneck or availability risk. The EKS target platform (ADR-001) must receive live traffic before the legacy system is decommissioned
- **Regulatory context**: PCI-DSS v4.0 Requirement 1.3 mandates network access controls and segmentation for the cardholder data environment; the routing layer is the natural enforcement point. UK GDPR Article 44 data residency requires all order processing traffic to remain within eu-west-2 (primary) / eu-west-1 (DR)

## 5. Considered Options

### Option A: Strangler-Fig with Amazon API Gateway (Chosen)

**Description**: Deploy Amazon API Gateway as a Regional HTTP façade in eu-west-2. All inbound order management traffic (REST API calls from web, mobile, partner, and internal channels) routes through API Gateway. A route table maps URL path patterns to either the legacy monolith (via VPC Link to legacy NLB) or the new EKS microservices (via VPC Link to the EKS internal ALB). As each microservice is migrated, validated, and load-tested, its route is redirected from legacy to EKS. Traffic migration is complete when all routes point to EKS and the legacy platform is fully decommissioned.

**Implementation approach**:

1. Month 1 — Deploy API Gateway Regional endpoint with all routes → legacy (transparent proxy). No observable change to users. Deploy WAF with OWASP Core Rule Set. Establish CDK stack. Migrate DNS to custom domain with ACM certificate.
2. Months 2–4 — Migrate read-heavy services first (Order Status, Order Search, Order History). Canary each route at 5% / 95% split for minimum 48 hours before full redirect.
3. Months 5–7 — Migrate PCI-DSS scoped routes (card tokenisation, payment processing, payment reconciliation). WAF rules enforce CDE boundary. QSA evidence pack assembled.
4. Month 7 — QSA audit submission. All high-severity findings evidenced as closed.
5. Months 8–18 — Migrate remaining order management routes (order placement, fulfillment, dispatch, returns, reconciliation, reporting). Legacy scaled down as routes drain.
6. Months 19–24 — Legacy decommissioned. API Gateway sole routing layer.

**Route management governance**: All route definitions in CDK TypeScript (`/infrastructure/api-gateway/`). Route changes require PR review with checklist: staging tested, rollback documented and drilled, error rate baseline established.

**Wardley Evolution Stage**: Product (Amazon API Gateway is a widely-adopted, mature managed service with defined SLA)

**Pros**:
- ✅ Satisfies BR-001 zero-downtime constraint — per-route redirect is instantaneous and independently reversible
- ✅ Enables BR-004 PCI-DSS scope isolation — WAF + VPC Link creates auditable CDE boundary before Month 7
- ✅ Per-route rollback in < 5 minutes — API Gateway stage variable update, no code change, no DNS TTL wait
- ✅ API Gateway SLA 99.95% — exceeds NFR-A-001 target of 99.9%
- ✅ WAF integration — OWASP Core Rule Set, rate limiting, IP reputation; DDoS protection via AWS Shield Standard
- ✅ Canary deployment support — weighted routing (e.g., 5% / 95%) for staged traffic shifts reduces R-013
- ✅ Strategic asset beyond migration — all future channels (B2B portal, partner APIs) connect via the same façade (BR-007, BR-008)
- ✅ Access logs → CloudWatch → SIEM — PCI-DSS audit trail requirement satisfied

**Cons**:
- ❌ Adds 10–30 ms routing latency per request; must be budgeted within NFR-P-001 (3,000 ms)
- ❌ Dual-run costs: legacy infrastructure must remain operational during migration period (~18 months)
- ❌ VPC Link setup complexity: requires NLB in each VPC; adds configuration and NLB running costs
- ❌ API Gateway REST API payload limit: 10 MB maximum — acceptable for order management (typical < 50 KB), but restricts future bulk operations

### Option B: Big-Bang Parallel Platform Migration

**Description**: Build the complete new EKS-based platform in parallel with the legacy system. Replicate all order data in real time from legacy to Aurora PostgreSQL using DMS CDC. At a predetermined maintenance window (maximum 2 hours), disable the legacy system, validate data consistency, and redirect all DNS to the new platform. If validation fails within the window, revert DNS to legacy.

**Implementation approach**: Full EKS platform build (Months 1–9). DMS CDC replication from legacy database (Months 3–9). Parallel validation with production data snapshots (Month 9). Hard cutover maintenance window (Month 10, 2-hour planned window). Legacy decommission (Months 11–12).

**Wardley Evolution Stage**: Custom-Built (cutover orchestration tooling, data consistency validation scripts, and rollback automation are bespoke for this migration)

**Pros**:
- ✅ No routing layer complexity — simpler operational model after cutover
- ✅ Shorter dual-run period (legacy decommissioned Month 10–12 vs. Month 19–24)
- ✅ Clean architectural break — no legacy routing dependency post-cutover

**Cons**:
- ❌ **Directly violates BR-001** — requires minimum 2-hour planned downtime; unacceptable for a revenue-critical system processing 600+ orders/minute
- ❌ **PCI-DSS timeline structurally incompatible** — full parallel platform build cannot complete before Month 7 QSA submission deadline; BR-004 fails
- ❌ **Higher blast radius** — 100% of order volume affected if cutover fails; no partial rollback
- ❌ **R-014 substantially elevated** — full data consistency validation at a single point in time carries higher data integrity risk than per-service validation
- ❌ **R-013 substantially elevated** — DNS rollback takes 5–60 minutes (TTL propagation) vs. < 5 minutes for route variable revert
- ❌ No early value delivery — investment locked until Month 10 cutover; no migrated services delivering value earlier

## 6. Decision Outcome

### 6.1 Chosen Option

**Option A: Strangler-Fig with Amazon API Gateway**

### 6.2 Y-Statement

> In the context of incrementally migrating the Legacy Order Management System to cloud-native microservices on Amazon EKS across a 24-month programme,
> facing the constraint that customer-facing order processing cannot tolerate planned downtime and PCI-DSS scope isolation must be established by Month 7,
> we decided to adopt the strangler-fig migration pattern with Amazon API Gateway as the traffic façade,
> to achieve zero-downtime per-route migration with < 5-minute rollback capability, PCI-DSS network scope isolation at the routing layer, and a reusable API façade asset for all future channel integrations,
> accepting the additional 10–30 ms routing latency (within NFR-P-001 budget), the dual-run infrastructure cost of ~£90,000 over the 18-month migration period, and the operational overhead of managing API Gateway stage configuration and WAF rules.

### 6.3 Justification

**Why not Option B (Big-bang)?**

Option B is eliminated because it directly violates BR-001 (zero unplanned downtime — MUST_HAVE, no waiver authority). The 2-hour maintenance window is commercially unacceptable for a revenue-critical system. In addition, the PCI-DSS re-certification deadline (Month 7 QSA submission) is structurally incompatible with a parallel full-platform build timeline: the full EKS platform cannot be built and validated in time. R-013 (customer-visible service degradation) and R-014 (order data loss) residual scores are substantially higher under Option B than Option A due to the single high-risk cutover event and slow DNS rollback.

**Why Option A?**

- The strangler-fig pattern is recommended in `ARC-001-SOBC-v1.0` (Option 2, preferred), `ARC-001-AWRS-v1.0` (Section 3), and is the canonical implementation of Principle 1 (Incremental Modernization) in `ARC-000-PRIN-v1.0`
- Amazon API Gateway is confirmed in `ARC-001-AWRS-v1.0` as the recommended strangler-fig façade, offering a 99.95% availability SLA, Regional Multi-AZ deployment, WAF integration, and CDK-native configuration
- Per-route rollback in < 5 minutes (stage variable update) vs. > 30 minutes DNS TTL for Option B — a material difference in R-013 risk
- API Gateway generates strategic value beyond migration: all future channels (B2B portal per BR-007, logistics partner integration per BR-008) connect via the same façade (Principle 2: API-First Design)
- Canary deployment support (weighted routing) reduces R-013 further by enabling 5% / 95% traffic splits during each phase transition

**Stakeholder consensus**:
- CISO confirmed API Gateway WAF satisfies PCI-DSS Requirement 1.3 network segmentation
- VP Operations confirmed zero-downtime constraint is non-negotiable (SD-4 primary driver)
- CTO endorsed AWS managed service preference per Principle 8; dual-run cost accepted vs. big-bang risk premium

## 7. Consequences

### 7.1 Positive Consequences

- **Zero-downtime migration (BR-001)**: Each service migrates independently via route redirection. Measured outcome target: 0 customer-impacting incidents attributable to migration activity across all phases.
- **PCI-DSS scope isolation (BR-004)**: API Gateway WAF + VPC Link creates an auditable network boundary for the cardholder data environment. Target: QSA audit submitted Month 7; all 3 high-severity findings evidenced as closed; re-certification letter received Month 8. Baseline: 3 unresolvable findings → target: 0.
- **< 5-minute rollback**: Stage variable update immediately restores legacy routing for any route. Reduces R-013 (service degradation) and R-014 (data loss) residual risk scores. Rollback RTO: baseline ~30 min (DNS) → target < 5 min (stage variable).
- **API façade as strategic asset**: All future channel integrations (B2B portal, partner webhook APIs, mobile) connect via the same API Gateway. Supports BR-007 (real-time order status) and BR-008 (24-hour order-to-ship with real-time logistics APIs) without additional routing infrastructure.
- **Daily deployments enabled (BR-006)**: Once a service route is migrated to EKS, it deploys independently of the legacy platform. Deployment frequency: legacy ~quarterly → EKS ~daily from Month 12. Change failure rate target: below 10% from Month 12.
- **Canary deployment support**: Weighted routing (5% / 95%) enables staged traffic shifts per route, reducing blast radius of each migration step.
- **Metrics baseline → target**:

| Metric | Baseline | Target | Timeline |
|--------|----------|--------|----------|
| Unplanned outages from migration | 0 | 0 (sustained) | All phases |
| Rollback time | ~30 min (DNS TTL) | < 5 min (stage variable) | Phase 0 |
| PCI-DSS high-severity findings | 3 | 0 | Month 8 |
| Routes on EKS | 0% | 100% | Month 18 |
| Legacy decommission | Ongoing | Complete | Month 24 |

### 7.2 Negative Consequences (Accepted Trade-offs)

- **Routing latency overhead (10–30 ms)**: API Gateway adds one managed network hop per request. At NFR-P-001 p95 target of < 3,000 ms for order submission (typical backend processing 200–500 ms), the 30 ms overhead consumes less than 5% of the latency budget. Mitigation: Regional endpoint (not Edge-optimised) minimises latency; CloudWatch alert fires at p95 > 2,500 ms (500 ms buffer before NFR breach); monitored from Phase 0.
- **Dual-run infrastructure cost (~£90,000 over 18 months)**: Legacy infrastructure must remain operational until all routes are drained. Mitigation: scale down legacy components as corresponding routes migrate; target 50% legacy cost reduction by Month 12 as PCI-DSS scoped components migrate first (highest infrastructure cost).
- **API Gateway operational complexity**: Stage management, route versioning, WAF rule tuning, VPC Link NLB health checks. Mitigation: all configuration in CDK TypeScript with PR review; runbooks authored before Phase 1; R-006 (skills gap) treatment plan includes API Gateway certification training for 2 engineers.
- **10 MB payload limit (REST API)**: API Gateway REST API imposes a 10 MB maximum request payload. Typical order management payloads are < 50 KB. If bulk operations with larger payloads are required in future, the HTTP API Gateway variant or direct ALB access may be required for those endpoints only.

### 7.3 Neutral Consequences (Changes Required)

- **New CDK stack**: `/infrastructure/api-gateway/` CDK TypeScript stack covering API Gateway Regional endpoint, WAF web ACL, VPC Links, NLBs, custom domain, and ACM certificate.
- **DNS migration**: All channel DNS records updated from legacy load balancer to API Gateway custom domain in Month 1 (low-risk: transparent proxy initially). Domain team action required.
- **Legacy NLB adapter**: Legacy monolith must expose a VPC-routable NLB target endpoint for API Gateway VPC Link. Estimated 1-week engineering effort in Month 1.
- **Monitoring update**: CloudWatch dashboards extended with per-route request count, error rate, latency percentiles. Operations teams trained on new dashboards before Phase 1.
- **Phase gate process**: Each route migration requires documented entry criteria (EKS service deployed, load-tested at NFR-P-002 throughput, security reviewed by CISO) and exit criteria (error rate < 0.1% sustained for 2 weeks post-route migration).

### 7.4 Risks and Mitigations

| Risk ID | Risk | Likelihood | Impact | Mitigation | Owner |
|---------|------|------------|--------|------------|-------|
| R-013 | Customer-visible service degradation during route migration | Medium | High | Canary deployments (5% / 95% split minimum 48 hours); per-route circuit breaker; < 5 min rollback via stage variable | VP Operations |
| R-014 | Order data loss during migration | Low | High | DMS CDC dual-write validation before each route migration; per-service data consistency verification gate | CTO / EA |
| R-001 | Programme scope creep extending migration timeline | Medium | Medium | Phase gates with documented entry/exit criteria; SCCB approval required for scope changes | Executive Sponsor |
| R-004 | Engineering capacity dragged by legacy maintenance | High | Medium | Route-by-route migration progressively reduces legacy surface; engineers freed as routes migrate | CTO / CIO |
| NEW-001 | API Gateway service quota exhaustion at peak load | Low | High | Raise service quota to 10,000 RPS before Phase 1 (default 10,000 RPS sufficient for NFR-P-002 peak); CloudWatch alarm at 80% quota utilisation | Engineering Lead |
| NEW-002 | VPC Link NLB health check misconfiguration causes false routing failures | Low | Medium | CDK unit tests for NLB target group configuration; full staging environment validation before each phase transition | Engineering Lead |
| NEW-003 | WAF false-positive blocks legitimate order API traffic | Low | Medium | WAF deployed in Count mode (monitoring only) for 2 weeks before switching to Block mode; alert on 4XX rate > 5% | CISO / Engineering Lead |

### Appendix B: API Gateway Architecture

All API Gateway configuration is managed via AWS CDK TypeScript. The routing infrastructure comprises the following components:

**Endpoint configuration**: Regional REST API in eu-west-2 with a custom domain backed by an ACM certificate. Regional endpoint type is chosen over Edge-optimised to minimise latency for UK-based clients and to maintain data residency within eu-west-2.

**Traffic routing mechanism**: VPC Links connect the API Gateway to internal Network Load Balancers in both the legacy VPC and the EKS VPC. Stage variables define the active VPC Link target per route. A stage variable update redirects a route from legacy to EKS without requiring a full API Gateway redeploy — this enables the < 5-minute rollback target.

**Security controls**: A WAF Web ACL is attached to the API Gateway stage. The ACL includes the AWS Managed Rules OWASP Core Rule Set, AWS Managed Rules Known Bad Inputs rule group, and a rate limiting rule scoped to order creation endpoints. All WAF rules are deployed in Count mode for 2 weeks before switching to Block mode to establish a false-positive baseline.

**Logging**: Structured JSON access logs are written to CloudWatch Logs for every route. Log groups are retained for 90 days (PCI-DSS audit trail requirement). Log entries include method, resource path, HTTP status, integration latency, and caller identity — but exclude request and response body content (GDPR data minimisation).

**CDK deployment**: The API Gateway stack is deployed via the CI/CD pipeline (Principle 21: CI/CD Automation). All route changes, WAF rule updates, and stage variable changes are CDK-managed, PR-reviewed, and deployed through the pipeline — no manual console changes are permitted.
