## 3. コンテキストと問題の定義

### 3.1 問題の説明

Legacy Order Management System（OMS）は、ADR-001 に基づき Amazon EKS 上のクラウドネイティブなマイクロサービスアーキテクチャに移行しなければならないモノリシックアプリケーションである。組織は Web、モバイル、パートナー、B2B の各チャネルを通じて継続的に注文を処理している。計画停止ウィンドウや一括カットオーバーは、平均 600 注文/分・ピーク 1,800 注文/分（NFR-P-002）を処理する収益重要システムには商業的に受け入れられない。インバウンドリクエストをレガシープラットフォームから EKS ベースの新規マイクロサービスへ段階的に移行する方法を規定するマイグレーションパターンとトラフィックルーティング機構について決定が必要である。

**問題の定義（問いの形式）**: ゼロダウンタイムの事業継続性を維持し、ルート単位の即時ロールバックを実現し、Month 7 までに PCI-DSS スコープ分離を施行しながら、レガシーモノリスから EKS 上のクラウドネイティブマイクロサービスへ受注管理トラフィックを段階的に移行するためのマイグレーションパターンはどれか？

### 3.2 この決定が必要な理由

プログラムは、単一の高リスクカットオーバーイベントなしにモノリスからマイクロサービスへ橋渡しをしなければならない（〜しなければならない）。この決定は、24 か月プログラム全体を規定するトラフィックルーティング機構、ロールバック戦略、PCI-DSS スコープ境界、およびマイグレーションシーケンスフレームワークを確立する。

- **ビジネスコンテキスト**: BR-001 は各フェーズでの計画外ゼロダウンタイムによる段階的マイグレーションを義務付ける；BR-004 は Month 8（固定の外部期限、レガシープラットフォームでは解決不能）までの PCI-DSS スコープ分離と再認定を要求；BR-003 は Month 12 から 99.9% 稼働率を要求
- **技術コンテキスト**: NFR-A-001（Month 12 から月次稼働率 99.9%）、NFR-P-001（注文送信 p95 < 3 秒）、NFR-P-002（1,800 注文/分ピークスループット）— ルーティング層はボトルネックや可用性リスクになってはならない（〜してはならない）。EKS ターゲットプラットフォーム（ADR-001）はレガシーシステムが廃止される前にライブトラフィックを受ける必要がある
- **規制コンテキスト**: PCI-DSS v4.0 要件 1.3 はカードホルダーデータ環境のネットワークアクセス制御とスコープ分割を義務付ける；ルーティング層は自然な施行ポイントとなる。UK GDPR Article 44 のデータレジデンシー要件はすべての注文処理トラフィックを eu-west-2（プライマリ）/ eu-west-1（DR）内に留めることを要求する

## 5. 検討した選択肢

### 選択肢 A: Strangler-Fig + Amazon API Gateway（採用）

**説明**: eu-west-2 に Amazon API Gateway の Regional HTTP ファサードをデプロイする。すべての受信受注管理トラフィック（Web、モバイル、パートナー、内部チャネルからの REST API 呼び出し）は API Gateway を経由する。ルートテーブルは URL パスパターンをレガシーモノリス（VPC Link 経由でレガシー NLB へ）または新規 EKS マイクロサービス（VPC Link 経由で EKS 内部 ALB へ）のどちらかにマッピングする。各マイクロサービスが移行・検証・負荷テストを経て、そのルートをレガシーから EKS に切り替える。すべてのルートが EKS を指しレガシープラットフォームが完全に廃止された時点でトラフィック移行が完了する。

**実装アプローチ**:

1. Month 1 — API Gateway Regional エンドポイントをデプロイし、すべてのルートをレガシーに向ける（透過プロキシ）。ユーザーからは変化なし。OWASP Core Rule Set を持つ WAF をデプロイ。CDK スタックを構築。ACM 証明書によるカスタムドメインに DNS を移行。
2. Month 2〜4 — 読み取り負荷の高いサービスを最初に移行（注文ステータス、注文検索、注文履歴）。各ルートを 5% / 95% スプリットで最低 48 時間カナリア実施してから完全切り替え。
3. Month 5〜7 — PCI-DSS スコープルートを移行（カードトークナイゼーション、決済処理、決済照合）。WAF ルールが CDE 境界を施行。QSA エビデンスパックを編集。
4. Month 7 — QSA 監査提出。高深刻度指摘事項はすべてクローズ済みとしてエビデンス。
5. Month 8〜18 — 残りの受注管理ルートを移行（注文受付、フルフィルメント、出荷、返品、照合、レポート）。ルートが空になるにつれてレガシーをスケールダウン。
6. Month 19〜24 — レガシー廃止。API Gateway が唯一のルーティング層となる。

**ルート管理ガバナンス**: すべてのルート定義を CDK TypeScript（`/infrastructure/api-gateway/`）で管理。ルート変更には PR レビューとチェックリストが必要: ステージングでテスト済み、ロールバックが文書化・演練済み、エラーレートのベースラインが確立済み。

**Wardley 進化段階**: 製品（Amazon API Gateway は定義された SLA を持つ広く採用された成熟したマネージドサービス）

**利点（Pros）**:
- ✅ BR-001 ゼロダウンタイム制約を満たす — ルート単位のリダイレクトは即時かつ独立してリバーシブル
- ✅ BR-004 PCI-DSS スコープ分離を実現 — WAF + VPC Link が Month 7 前に監査可能な CDE 境界を作成
- ✅ 5 分未満のルート単位ロールバック — API Gateway ステージ変数の更新、コード変更なし、DNS TTL 待ち不要
- ✅ API Gateway SLA 99.95% — NFR-A-001 の 99.9% 目標を超過
- ✅ WAF 統合 — OWASP Core Rule Set、レート制限、IP レピュテーション；AWS Shield Standard による DDoS 保護
- ✅ カナリアデプロイサポート — 重み付けルーティング（例: 5% / 95%）による段階的なトラフィック移行で R-013 を低減
- ✅ マイグレーション後も継続的に活用できる戦略的資産 — 将来のすべてのチャネル（B2B ポータル、パートナー API）が同じファサード経由で接続（BR-007、BR-008）
- ✅ アクセスログ → CloudWatch → SIEM — PCI-DSS 監査証跡要件を満たす

**欠点（Cons）**:
- ❌ リクエストごとに 10〜30 ms のルーティングレイテンシーが加算；NFR-P-001（3,000 ms）の予算内に収める必要がある
- ❌ 並行稼働コスト: マイグレーション期間中（約 18 か月）レガシーインフラを稼働し続ける必要がある
- ❌ VPC Link セットアップの複雑性: 各 VPC に NLB が必要；構成と NLB 稼働コストが増加する
- ❌ API Gateway REST API ペイロード制限: 最大 10 MB — 受注管理には許容範囲（通常 < 50 KB）だが、将来のバルク操作を制約する

## 6. 意思決定の結果

### 6.1 選択した選択肢

**選択肢 A: Strangler-Fig + Amazon API Gateway**

### 6.2 Y ステートメント

> 24 か月プログラムにわたって Legacy Order Management System を Amazon EKS 上のクラウドネイティブマイクロサービスに段階的に移行するという状況において、
> 顧客向け注文処理は計画停止を許容できず、PCI-DSS スコープ分離を Month 7 までに確立しなければならないという制約に直面しつつ、
> Amazon API Gateway をトラフィックファサードとした Strangler-fig マイグレーションパターンを採用することを決定した。
> 5 分未満のロールバック機能を持つルート単位のゼロダウンタイムマイグレーション、ルーティング層における PCI-DSS ネットワークスコープ分離、および将来のすべてのチャネル統合に向けた再利用可能な API ファサード資産を実現するために、
> 追加の 10〜30 ms ルーティングレイテンシー（NFR-P-001 予算内）、18 か月のマイグレーション期間にわたる約 £90,000 の並行稼働インフラコスト、API Gateway のステージ構成と WAF ルールの運用管理オーバーヘッドを受容する。

### 6.3 正当化

**なぜ選択肢 B（ビッグバン）ではないのか？**

選択肢 B は BR-001（計画外ゼロダウンタイム — MUST_HAVE、免除権限なし）に直接違反するため除外される。2 時間のメンテナンスウィンドウは収益重要システムには商業的に受け入れられない。さらに、PCI-DSS 再認定期限（Month 7 の QSA 提出）は並行フルプラットフォーム構築タイムラインとは構造的に非互換である: 完全な EKS プラットフォームを時間内に構築・検証することができない。R-013（顧客可視のサービス劣化）と R-014（注文データ損失）の残余スコアは、単一の高リスクカットオーバーイベントと低速な DNS ロールバックにより、選択肢 A と比較して選択肢 B では大幅に高い。

**なぜ選択肢 A なのか？**

- Strangler-fig パターンは `ARC-001-SOBC-v1.0`（選択肢 2、推奨）、`ARC-001-AWRS-v1.0`（Section 3）で推奨されており、`ARC-000-PRIN-v1.0` の原則 1（段階的モダナイゼーション）の標準実装である
- Amazon API Gateway は `ARC-001-AWRS-v1.0` で Strangler-fig ファサードとして推奨されており、99.95% 可用性 SLA、Regional マルチ AZ デプロイ、WAF 統合、CDK ネイティブな構成を提供することが確認されている
- 5 分未満のルート単位ロールバック（ステージ変数更新）vs. 選択肢 B の 30 分超の DNS TTL — R-013 リスクにおいて重要な差異
- API Gateway はマイグレーションを超えた戦略的価値を生み出す: 将来のすべてのチャネル（BR-007 の B2B ポータル、BR-008 の物流パートナー統合）が同じファサード経由で接続する（原則 2: API ファースト設計）
- カナリアデプロイサポート（重み付けルーティング）が各フェーズ遷移の影響範囲を限定することで R-013 をさらに低減

**ステークホルダーのコンセンサス**:
- CISO が API Gateway WAF は PCI-DSS 要件 1.3 のネットワーク分割を満たすと確認
- VP Operations がゼロダウンタイム制約は非交渉事項であることを確認（SD-4 の主要ドライバー）
- CTO が原則 8 に基づく AWS マネージドサービス優先を支持；ビッグバンのリスクプレミアムと比較して並行稼働コストを受容；API Gateway をエンタープライズ API 標準として承認

## 7. 結果と影響

### 7.1 肯定的な結果

- **ゼロダウンタイムマイグレーション（BR-001）**: 各サービスがルートリダイレクトにより独立してマイグレーション。目標測定アウトカム: すべてのフェーズにわたりマイグレーション活動に起因する顧客影響インシデント 0 件。
- **PCI-DSS スコープ分離（BR-004）**: API Gateway WAF + VPC Link がカードホルダーデータ環境の監査可能なネットワーク境界を作成。目標: Month 7 に QSA 監査提出；3 件の高深刻度指摘事項がすべてクローズ済みとしてエビデンス；Month 8 に再認定書受領。ベースライン: 3 件の解決不能な指摘事項 → 目標: 0 件。
- **5 分未満のロールバック**: ステージ変数更新でいかなるルートも即座にレガシールーティングを復元。R-013（サービス劣化）と R-014（データ損失）の残余リスクスコアを低減。ロールバック RTO: ベースライン 約 30 分（DNS）→ 目標 5 分未満（ステージ変数）。
- **戦略的資産としての API ファサード**: 将来のすべてのチャネル統合（B2B ポータル、パートナー Webhook API、モバイル）が同じ API Gateway 経由で接続。追加のルーティングインフラなしに BR-007（リアルタイム注文ステータス）と BR-008（24 時間注文出荷・リアルタイム物流 API）をサポート。
- **日次デプロイの実現（BR-006）**: サービスルートが EKS に移行すると、レガシープラットフォームとは独立してデプロイされる。デプロイ頻度: レガシー 約四半期ごと → Month 12 から EKS 日次。変更失敗率目標: Month 12 から 10% 未満。
- **カナリアデプロイサポート**: 重み付けルーティング（5% / 95%）により、ルートごとの段階的なトラフィック移行が可能で、各マイグレーションステップの影響範囲を限定。
- **指標のベースライン → 目標**:

| 指標 | ベースライン | 目標 | タイムライン |
|--------|----------|--------|----------|
| マイグレーションによる計画外停止 | 0 | 0（維持） | 全フェーズ |
| ロールバック時間 | 約 30 分（DNS TTL） | 5 分未満（ステージ変数） | Phase 0 |
| PCI-DSS 高深刻度指摘事項 | 3 | 0 | Month 8 |
| EKS 上のルート | 0% | 100% | Month 18 |
| レガシー廃止 | 継続中 | 完了 | Month 24 |

### 7.2 否定的な結果（受容済みトレードオフ）

- **ルーティングレイテンシーオーバーヘッド（10〜30 ms）**: API Gateway はリクエストごとに 1 つのマネージドネットワークホップを追加する。NFR-P-001 の注文送信 p95 目標 < 3,000 ms（典型的なバックエンド処理 200〜500 ms）において、30 ms のオーバーヘッドはレイテンシー予算の 5% 未満を消費する。軽減策: Regional エンドポイント（Edge 最適化ではない）でレイテンシーを最小化；p95 > 2,500 ms で CloudWatch アラート（NFR 違反まで 500 ms のバッファー）；Phase 0 からモニタリング。
- **並行稼働インフラコスト（18 か月で 約 £90,000）**: すべてのルートが空になるまでレガシーインフラを稼働し続ける必要がある。軽減策: 対応するルートが移行するにつれてレガシーコンポーネントをスケールダウン；最もインフラコストが高い PCI-DSS スコープコンポーネントが最初に移行することで Month 12 までに 50% のレガシーコスト削減を目標とする。
- **API Gateway 運用の複雑性**: ステージ管理、ルートバージョニング、WAF ルールチューニング、VPC Link NLB ヘルスチェック。軽減策: すべての構成を PR レビューで CDK TypeScript に記述；Phase 1 前にランブックを作成；R-006（スキルギャップ）対応計画に 2 名のエンジニアへの API Gateway 認定トレーニングを含める。
- **10 MB ペイロード制限（REST API）**: API Gateway REST API は最大 10 MB のリクエストペイロードを課す。典型的な受注管理ペイロードは < 50 KB。将来より大きなペイロードを持つバルク操作が必要になった場合、それらのエンドポイントにのみ HTTP API Gateway バリアントまたは直接 ALB アクセスが必要となる可能性がある。

### 7.3 中立的な結果（必要な変更）

- **新規 CDK スタック**: API Gateway Regional エンドポイント、WAF Web ACL、VPC Link、NLB、カスタムドメイン、ACM 証明書をカバーする `/infrastructure/api-gateway/` CDK TypeScript スタック。
- **DNS マイグレーション**: すべてのチャネル DNS レコードを Month 1 にレガシーロードバランサーから API Gateway カスタムドメインに更新（低リスク: 初期は透過プロキシ）。ドメインチームの対応が必要。
- **レガシー NLB アダプター**: レガシーモノリスは API Gateway VPC Link のために VPC ルーティング可能な NLB ターゲットエンドポイントを公開しなければならない（〜しなければならない）。Month 1 での推定 1 週間のエンジニアリング工数。
- **モニタリング更新**: ルートごとのリクエスト数、エラーレート、レイテンシーパーセンタイルで CloudWatch ダッシュボードを拡張。Phase 1 前に運用チームが新しいダッシュボードの研修を受ける。
- **フェーズゲートプロセス**: 各ルートマイグレーションには文書化されたエントリー基準（EKS サービスのデプロイ、NFR-P-002 スループットでの負荷テスト、CISO によるセキュリティレビュー）と終了基準（ルートマイグレーション後 2 週間継続してエラーレート < 0.1%）が必要。

**GDS Service Standard**:

- ポイント 9（セキュアなサービスを作る）: WAF 構成を CISO がレビューし署名；PCI-DSS スコープ分離を Month 7 の QSA 監査パックでエビデンス；ペネトレーションテスト結果
- ポイント 14（信頼性の高いサービスを運用する）: 99.95% API Gateway SLA のエビデンス；フェーズごとにロールバック演練を文書化

**Technology Code of Practice**:
- ポイント 5（クラウドファースト）: Amazon API Gateway マネージドサービスをセルフマネージドプロキシより選択 — パッチ当てが必要なサーバーなし
- ポイント 8（オープンにする）: すべての API Gateway 構成を CDK TypeScript に記述し、プロジェクトリポジトリでバージョン管理

**セキュリティ保証**:
- OWASP Core Rule Set を WAF にデプロイし、Block モード有効化前に 2 週間 Count モードで観察
- AWS Shield Standard が API Gateway に含まれる；Shield Advanced を PCI-DSS スコープフェーズで評価
- API Gateway と EKS 内部 ALB の間で mTLS を設定（相互証明書認証、片方向 TLS のみではない）
- API Gateway 管理 API は VPC エンドポイント経由でアクセス — コントロールプレーンはインターネットに公開しない
- アクセスログ: 構造化 JSON を CloudWatch Logs に書き込み；ロググループは PCI-DSS コンプライアンスのために 90 日間保持

**データ保護（UK GDPR）**:
- すべてのトラフィックルートは eu-west-2（プライマリ）/ eu-west-1（DR）内のみ — Article 44 データレジデンシー要件を満たす
- API Gateway アクセスログはリクエストメタデータ（メソッド、パス、ステータス、レイテンシー、ソース IP）を記録するが、注文ペイロードの内容は除外 — Article 5(1)(c) のデータ最小化を満たす
- カナリアトラフィックスプリットは個人データをテストまたは非本番環境にルーティングしない — 本番ルーティングのみ

### 付録 B: API Gateway アーキテクチャ

すべての API Gateway 構成は AWS CDK TypeScript で管理する。ルーティングインフラは以下のコンポーネントで構成される。

**エンドポイント構成**: eu-west-2 に ACM 証明書でバックアップされたカスタムドメインを持つ Regional REST API。UK ベースのクライアントへのレイテンシーを最小化し、eu-west-2 内のデータレジデンシーを維持するために、Edge 最適化ではなく Regional エンドポイントタイプを選択する。

**トラフィックルーティング機構**: VPC Link がレガシー VPC と EKS VPC の両方にある内部 Network Load Balancer に API Gateway を接続する。ステージ変数がルートごとのアクティブな VPC Link ターゲットを定義する。ステージ変数の更新は API Gateway の完全な再デプロイを必要とせずにルートをレガシーから EKS に切り替える — これにより 5 分未満のロールバック目標が実現される。

**セキュリティ統制**: WAF Web ACL が API Gateway ステージに付与される。ACL には AWS マネージドルール OWASP Core Rule Set、AWS マネージドルール既知の不正入力ルールグループ、および注文作成エンドポイントにスコープされたレート制限ルールが含まれる。すべての WAF ルールは誤検知ベースラインを確立するために Block モードに切り替える前の 2 週間 Count モードでデプロイする。

**ログ**: すべてのルートのリクエストごとに構造化 JSON アクセスログを CloudWatch Logs に書き込む。ロググループは 90 日間保持（PCI-DSS 監査証跡要件）。ログエントリにはメソッド、リソースパス、HTTP ステータス、統合レイテンシー、呼び出し元 ID を含むが、リクエストおよびレスポンスのボディ内容は除外する（GDPR データ最小化）。

**CDK デプロイ**: API Gateway スタックは CI/CD パイプライン経由でデプロイする（原則 21: CI/CD 自動化）。すべてのルート変更、WAF ルール更新、ステージ変数変更は CDK 管理、PR レビュー済みで、パイプライン経由でデプロイする — 手動コンソール変更は禁止。
