## エグゼクティブサマリー

### 目的

本文書は、レガシー受注管理システム近代化プログラムに関わるすべての主要ステークホルダーを特定し、それぞれの根本的なドライバー（動機・課題・懸念）を明確にする。また、それらのドライバーが測定可能なゴールにどう転化されるか、および成功を確認する事業成果を整理する。ステークホルダー → ドライバー → ゴール → 成果の完全なトレーサビリティを提供し、要求の優先付け・アーキテクチャ判断・コミュニケーション計画の基盤となる。

### 主な知見

本プログラムは強力なエグゼクティブスポンサーシップを有しており、財務・運用・戦略の各ドライバーが一点に収束している。その共通ゴールは、脆弱でコストの高いレガシープラットフォームを、イベント駆動型のクラウドネイティブ受注管理基盤に置き換えることである。主たる緊張関係は、CFOのコスト早期実現への意向と、運用チームの移行中サービス断に対するリスク回避姿勢の間に存在する。`ARC-000-PRIN-v1.0` のアーキテクチャ原則に沿った、二重稼働期間を伴う段階的なストラングラーフィグアプローチにより、最もリスクの高い受注処理コアに着手する前に早期の成果を届けることができ、この対立を解消できる。

### 重要成功要因

- エグゼクティブスポンサーが積極的に可視化されたスポンサーシップを維持し、二重稼働フェーズにおけるスコープクリープからプログラムを守る
- 運用チームが移行計画に積極的に参画する（承認のみでなく）。エッジケースに関するドメイン知識は代替不能である
- アンチコラプションレイヤーの設計は、レガシー連携作業が始まる前に承認を得る。これによりレガシーデータモデルの汚染を防ぐ
- 各移行フェーズゲートにおいて、顧客向けの注文ステータスおよびフルフィルメント性能を現状以上に維持する
- エンジニアリングチームは実装フェーズと並行してではなく、少なくとも1フェーズ前倒しでモダンスタックの再教育を受ける

### ステークホルダー整合スコア

**総合整合度**: 高

13のステークホルダーグループのうち12グループが、同一のプログラム成果に収束するドライバーを共有している。唯一の実質的な対立——デリバリー速度と移行リスク——は、解決策が明確に理解されている。プログラム目的に根本的に反対するステークホルダーグループは存在しない。

```text
                              関心度
                低                          高
          ┌──────────────────────┬──────────────────────────┐
          │                      │                          │
          │     満足を維持        │      密接に管理           │
     高   │                      │                          │
          │  • CISO              │  • エグゼクティブスポンサー│

          │  • 内部監査           │  • CTO / CIO             │
 影       │  • 規制機関           │  • VP オペレーション      │
 響       │                      │  • VP コマーシャル / 営業 │
 力       ├──────────────────────┼──────────────────────────┤
          │                      │                          │
          │       監視            │      情報提供             │
          │                      │                          │
     低   │  • HR / チェンジ管理  │  • 財務 / 経理チーム      │
          │                      │  • 受注フルフィルメント    │
          │                      │  • カスタマーサービス      │
          │                      │  • エンジニアリングチーム  │
          │                      │  • エンドカスタマー        │
          │                      │  • 物流パートナー          │
          └──────────────────────┴──────────────────────────┘
```

**象限の解釈:**

- **密接に管理** (影響力大・関心度大): すべてのフェーズゲートで積極的なエンゲージメントが必要な主要意思決定者
- **満足を維持** (影響力大・関心度小): 定期的な保証が必要な影響力を持つステークホルダー——軽視するとブロッカーになる
- **情報提供** (影響力小・関心度大): 信頼を維持し、運用上の知見を引き出すために定期的な情報共有が必要なステークホルダー
- **監視** (影響力小・関心度小): コンテキストが変化しない限り、最小限のエンゲージメントで対応

## ステークホルダードライバー分析

### SD-1: エグゼクティブスポンサー (CEO/COO) — 戦略的変革

**ステークホルダー**: エグゼクティブスポンサー (CEO または COO)

**ドライバー区分**: 戦略的

**ドライバーステートメント**: 新しいビジネスモデル（サブスクリプション、マルチチャネル、B2B ポータル）や最新のパートナーエコシステムとの連携を支援できないレガシー受注管理プラットフォームによって生じている競合上の不利を解消する。

**背景・コンテキスト**: 競合他社はすでにモダンな受注管理プラットフォームへ移行しており、レガシーシステムでは構造上サポート不可能な新しい商業モデル（オンライン購入・店舗受取、サブスクリプション段階課金、リアルタイムパートナー API）を展開している。計画されている市場再配置の 24 ヶ月前までにこの能力格差を埋めるよう、取締役会レベルからの圧力がある。

**ドライバー強度**: 最重要

**推進要因**:
- 単一の最終形態への一括切替ではなく、能力インクリメントを段階的に提供するフェーズドロードマップ
- 取締役会に進捗を示す早期の成果（例：リアルタイム注文追跡、セルフサービスポータル）

**阻害要因**:
- 受注管理コアが安定する前に隣接システム（ERP、CRM）へのスコープ拡大が起きること
- 運用チームのリスク回避姿勢が移行フェーズを 24 ヶ月ウィンドウ以上に遅延させること

**関連ステークホルダー**: CFO（コスト整合）、CTO（デリバリーの担い手）、VP コマーシャル（新機能の受益者）

---

### SD-2: CFO / 財務責任者 — コスト削減と ROI

**ステークホルダー**: CFO / 財務責任者

**ドライバー区分**: 財務的

**ドライバーステートメント**: レガシー受注管理プラットフォームのライセンス費、専用サポート契約、手動照合作業、インシデント対応費を含む総所有コスト (TCO) の増加傾向を抑制し、近代化投資に対して 24 ヶ月以内に定量化可能なリターンを生み出す。

**背景・コンテキスト**: レガシープラットフォームは市場相場を上回るベンダーサポート契約（オリジナルベンダーはすでに主流ではない）を抱え、老朽化したコードベースの維持に専門の外部委託を必要とし、財務チームの工数を消費する月間 1,200 件超の手動照合イベントを発生させている。CFO は、広範な業務効率化プログラムの一環として受注管理テクノロジーコストを 30% 削減することを約束している。

**ドライバー強度**: 最重要

**推進要因**:
- 固定ライセンスモデルに代わるクラウドネイティブ従量課金モデル
- 自動化された order-to-cash パイプラインによる手動照合の排除
- 段階的移行中のレガシーインフラ早期廃止（完全切替前から部分的な節約を実現）

**阻害要因**:
- 二重稼働期間のコスト（新旧プラットフォームの同時稼働）が、節約実現前に一時的に総支出を増加させる可能性
- 移行フェーズの遅延が二重稼働ウィンドウを延長し、ビジネスケースを侵食する

**関連ステークホルダー**: エグゼクティブスポンサー（ROI 説明責任）、VP オペレーション（運用コストドライバー）、エンジニアリング（デリバリー速度）

---

### SD-3: CTO / CIO — 技術近代化と人材

**ステークホルダー**: CTO / CIO

**ドライバー区分**: 戦略的

**ドライバーステートメント**: レガシーモノリシック受注管理システムを、モジュラーでクラウドネイティブかつイベント駆動型のアーキテクチャに置き換える。これにより、エンジニアリング人材の採用・定着を促進し、新機能の市場投入期間を短縮し、エンタープライズ顧客・パートナーに対する技術的信頼性を確立する。

**背景・コンテキスト**: エンジニアの離職率は年々増加しており、退職者がレガシースタックを主な理由として挙げている。受注管理システムに関わる職種の採用は、モダンな部分のエステートに関わる職種より 3〜4 ヶ月遅い。CTO の 3 ヶ年テクノロジー戦略は、Year 3 末までにすべてのティア 1 レガシーシステムを排除することを約束している。

**ドライバー強度**: 高

**推進要因**:
- 面接や会議でエンジニアが語れる、モダンでオープンスタンダードなアーキテクチャ
- 官僚的な遅延なくガバナンスを提供するアーキテクチャ決定記録 (ARC-000-PRIN-v1.0)
- ファーストクラスの成果物としての開発者ツール、CI/CD パイプライン、自動テストへの投資

**阻害要因**:
- チームが移行中もレガシーの維持に多大な時間を費やし、モダンビルドの容量が不足すること
- 短期的な利便性のためのアーキテクチャ決定が、新しいテクノロジースタックで同じ結合問題を再現してしまうこと

**関連ステークホルダー**: エンジニアリングチーム（直接の受益者）、VP オペレーション（デリバリー依存）、エグゼクティブスポンサー（戦略的整合）

### SD-4: VP オペレーション — 運用安定性とリスク

**ステークホルダー**: VP オペレーション

**ドライバー区分**: 運用 / リスク

**ドライバーステートメント**: レガシーシステムの障害、データ整合性の問題、手動ワークアラウンドによって繰り返し発生している運用上の混乱を排除する。同時に、近代化プログラム自体が現状より深刻なサービス断をもたらさないことを確保する。

**背景・コンテキスト**: レガシー受注管理システムは過去 12 ヶ月で 14 件の計画外障害が発生しており、そのうち 6 件は 2 時間超、1 件は約 4,200 件の顧客注文に影響する 8 時間障害だった。各障害により、オペレーションと財務の 6〜8 名が関与する手動復旧プロセスが発動する。VP オペレーションの個人 KPI にはシステム稼働率と注文エラー率が含まれており、いずれも年々悪化している。

**ドライバー強度**: 高

**推進要因**:
- 各フェーズ本番前にテスト済みの明示的なロールバック計画を伴う段階的移行（ARC-000-PRIN-v1.0 の原則 1 に整合）
- レガシー廃止前に新システムの安定性を確認できる、明確な終了基準を持つ二重稼働期間
- 新プラットフォームの自動モニタリングとアラート設定により、オペレーションにリアルタイム可視性を提供

**阻害要因**:
- フェーズ間の安定化期間をスキップするような圧縮された移行タイムライン
- 移行計画へのオペレーション参画が不十分——オペレーションはエッジケースを把握している

**関連ステークホルダー**: 受注フルフィルメントチーム（直接影響）、カスタマーサービス（顧客向けの結果）、CFO（障害コスト）

### SD-5: VP コマーシャル / 営業 — 売上成長と顧客維持

**ステークホルダー**: VP コマーシャル / 営業責任者

**ドライバー区分**: 財務的 / 顧客

**ドライバーステートメント**: 注文フルフィルメント失敗や劣悪な注文体験（不正確な追跡、出荷遅延、注文変更のセルフサービス不可）によって生じた売上損失と顧客チャーンを回復し、レガシープラットフォームがサポートできない商業モデルから新たな収益源を開拓する。

**背景・コンテキスト**: NPS 調査は受注管理を 2 番目に評価の低い顧客タッチポイントとして特定している。過去 12 ヶ月に注文エラーを経験した顧客の推定 8〜12% が 90 日以内に再注文していない。レガシーシステムに API レイヤーが存在しないため、商業チームは B2B 顧客ポータルとサブスクリプション提供を立ち上げられていない——この 2 つの施策の合算パイプライン価値は年間 £3.2M である。

**ドライバー強度**: 高

**推進要因**:
- セルフサービス顧客ポータルを実現するリアルタイム注文ステータス API
- 物流パートナー向けの Webhook ベースのイベント通知（バッチファイルの置き換え）
- モダンプラットフォームの設定可能な注文タイプによる新商業モデル（サブスクリプション、階層型 SLA）の有効化

**阻害要因**:
- 長期化する移行タイムラインが、商業チームの新規収益施策の立ち上げをブロックし続けること
- 切替フェーズ中の顧客向け品質低下が、既存の NPS 問題を悪化させるリスク

**関連ステークホルダー**: エンドカスタマー（受益者）、VP オペレーション（フルフィルメント依存）、CTO（デリバリー）

### SD-6: CISO / セキュリティ責任者 — リスク低減とコンプライアンス

**ステークホルダー**: CISO / 情報セキュリティ責任者

**ドライバー区分**: コンプライアンス / リスク

**ドライバーステートメント**: レガシー受注管理プラットフォームにおけるセキュリティ負債——時代遅れの認証機構、不十分な監査ログ、パッチ適用の遅延——を是正し、近代化されたプラットフォームが ARC-000-PRIN-v1.0 で定義されたゼロトラストアーキテクチャ原則に基づいて構築されることを確保する。

**背景・コンテキスト**: レガシープラットフォームに対する最新のペネトレーションテストで、深刻度「高」3 件・「中」11 件の所見が特定された。廃止済みのプラットフォームにベンダーセキュリティパッチを適用できないため、そのうち 4 件はレガシーシステムから移行しない限り是正不可能である。深刻度「高」の 2 件は、PCI-DSS スコープにおける監査ログの不十分さに関係する。CISO は、12 ヶ月以内にすべての深刻度「高」の所見を解消するという取締役会レベルのコミットメントを持つ。

**ドライバー強度**: 高

**推進要因**:
- 各移行フェーズゲートに組み込まれたセキュリティ・バイ・デザインレビュー（最終段階への後付けでない）
- フェーズ 1 のアーキテクチャベースラインにシークレット管理とゼロトラストサービス認証を最初から含める
- 新プラットフォームの初日からすべての注文状態変更の構造的な監査ログを記録する

**阻害要因**:
- スピード優先で設計された移行フェーズが、セキュリティコントロールを後のフェーズに先送りすること
- レガシーシステムとのアンチコラプションレイヤーが、設計が不十分な場合に新たな攻撃対象領域をもたらすこと

**関連ステークホルダー**: コンプライアンス責任者 / 法務（規制上の結果）、エンジニアリングチーム（実装）、内部監査（証跡の利用）

### SD-7: コンプライアンス責任者 / 法務 — 規制遵守

**ステークホルダー**: コンプライアンス責任者 / 法務顧問

**ドライバー区分**: コンプライアンス

**ドライバーステートメント**: 近代化されたプラットフォームが GDPR のデータ主体の権利（消去の権利を含む）、決済データ処理に関する PCI-DSS に準拠し、紛争解決および規制機関からの照会への対応に十分な、不変かつクエリ可能な監査証跡を維持することを確保する。

**背景・コンテキスト**: 14 ヶ月前に受け付けた GDPR 個人情報開示請求 (SAR) の対応に 47 日を要した。レガシーシステム内の 6 つのテーブルに注文履歴データが分散しており、統合されたクエリパスが存在しないためである。ICO はその後、改善の証跡を求め組織に連絡を取っている。PCI-DSS の再認証は 8 ヶ月以内に期限を迎えており、監査担当者は監査ログのギャップを重大な懸念として指摘している。

**ドライバー強度**: 高

**推進要因**:
- フェーズ 1 からデータモデルに設計された不変の注文監査証跡（ARC-000-PRIN-v1.0 の原則 8）
- GDPR 保持スケジュールに整合した自動化されたデータ保持・削除の強制
- すべての注文データフィールドに適用された明確なデータ分類（ARC-000-PRIN-v1.0 の原則 9）

**阻害要因**:
- 8 ヶ月以内の PCI-DSS 再認証期限が、準備状況に関わらず移行タイムラインに圧力をかけること
- 明確な System of Record のない複数サービス間での注文データ複製が、GDPR 消去の権利をより複雑にすること（ARC-000-PRIN-v1.0 の原則 10）

**関連ステークホルダー**: CISO（技術実装）、エンジニアリング（データモデル設計）、財務（order-to-cash 監査証跡）

### SD-8: 財務 / 経理チーム — 手動照合の排除

**ステークホルダー**: 財務 / 経理オペレーションチーム

**ドライバー区分**: 運用 / 財務的

**ドライバーステートメント**: レガシー受注管理システムと財務・ERP システム間のデータ不整合によって発生している月間 1,200 件超の手動照合イベントを排除し、照合対応に従事している約 2 FTE を解放する。

**背景・コンテキスト**: レガシーシステムは夜間バッチファイルを通じて ERP へ注文データをエクスポートしている。バッチ失敗、フォーマット不整合、重複レコードが日常的な照合作業を発生させている。財務スタッフ 2 名が毎月の約 80% の時間を照合例外処理に費やしていると推計されている。この状況は月次決算の締め作業を目標より 2〜3 日遅延させる原因にもなっている。

**ドライバー強度**: 中

**推進要因**:
- 夜間バッチに代わるリアルタイムイベント駆動型 ERP 連携（ARC-000-PRIN-v1.0 の原則 14）
- クロスシステムの不整合を排除する注文財務データの単一信頼源（原則 10）
- 例外アラート付きの自動照合レポーティング

**阻害要因**:
- ERP 連携の再設計が初期移行フェーズのスコープ外であり、照合のメリットが後のフェーズに先送りされる可能性
- UAT・テストへの財務チームの参加が月次決算サイクルによって制約されること

**関連ステークホルダー**: CFO（コスト削減）、受注フルフィルメントチーム（上流データ品質）、エンジニアリング（連携設計）

### SD-9: 受注フルフィルメントチーム — 使いやすさと作業負荷の軽減

**ステークホルダー**: 受注フルフィルメント オペレーションスタッフ

**ドライバー区分**: 運用 / 個人的

**ドライバーステートメント**: エラーが多く手動ステップの多いレガシーワークフローを、認知負荷を軽減し、再入力を最小化し、長年の保守延期によって蓄積されたシステム固有のワークアラウンドを不要にする、直感的で自動化された受注管理インターフェースに置き換える。

**背景・コンテキスト**: 受注フルフィルメントチームは現在、レガシーシステムの既知の不具合に対する 14 件の文書化されたワークアラウンドに依存しており、特定の決済タイプで発生する重複注文バグを防ぐための必須の手動ステップも含まれる。受注管理システムのスタッフオンボーディング期間は 6 週間で、その主な理由は文書化されていないワークアラウンドにある。このチームの高い離職率は、部分的にシステムへの不満に起因している。

**ドライバー強度**: 高

**推進要因**:
- フェーズ 1 から設計・UAT プロセスに参加するフルフィルメントチームメンバー
- 既存のワークアラウンドを文書化し、新プラットフォームで体系的に排除する（新技術で再現しない）
- 各フェーズ切替の 4 週間前（当日ではなく）に提供されるトレーニング

**阻害要因**:
- レガシーの不具合を異なる不慣れな不具合に置き換えた新システムが、フラストレーションをむしろ増大させる可能性
- チームが早期に参画しない場合、新システムが運用の実態を反映しない恐れ

**関連ステークホルダー**: VP オペレーション（説明責任）、カスタマーサービス（下流への影響）、エンジニアリング（使いやすさの設計）

---

### SD-10: エンジニアリング / 開発チーム — 開発者体験

**ステークホルダー**: ソフトウェアエンジニアリング・プラットフォームエンジニアリングチーム

**ドライバー区分**: 戦略的 / 個人的

**ドライバーステートメント**: 適切な CI/CD パイプライン、オブザーバビリティツール、明確なアーキテクチャ境界を備えたモダンで文書化されテスト可能なコードベースで作業し、廃止されたレガシースタックではなく、関連性の高いテクノロジーでキャリアを積みながら自信を持って機能を提供できるようにする。

**背景・コンテキスト**: レガシー受注管理コードベースには自動テストがなく、2 週間の手動リリースサイクルがあり、サポート終了から 5 年が経過したフレームワークバージョンで記述されている。エンジニアはデプロイに高い不安を感じており、ある最近のデプロイは自動ロールバックパスがないために 11 時間かけてロールバックされた。過去 8 ヶ月で 3 名のシニアエンジニアがレガシースタックを主な理由として退職している。

**ドライバー強度**: 中

**推進要因**:
- チームが新プラットフォーム上でビルドを開始する前に実装される自動テストとワンクリックロールバックを備えた CI/CD パイプライン
- 任意の制約ではなく設計決定の根拠をエンジニアに提供するアーキテクチャ決定記録 (ADR)
- フェーズ 1 開始時からのモダンツールとインフラストラクチャ・アズ・コード

**阻害要因**:
- レガシー維持と新規開発に分断されたエンジニア——コンテキストスイッチングが品質とモラル両方を侵食すること
- 新しいアーキテクチャが新技術で同様のモノリシックな結合を再現した場合、エンジニアのモチベーションが急速に崩壊する

**関連ステークホルダー**: CTO（戦略オーナー）、VP オペレーション（デリバリー依存）、HR（定着への影響）

---

### SD-11: エンドカスタマー — サービス品質と透明性

**ステークホルダー**: エンドカスタマー (B2C・B2B)

**ドライバー区分**: 顧客

**ドライバーステートメント**: 正確でタイムリーでセルフサービス可能な受注管理——リアルタイム追跡、遅延に関するプロアクティブな通知、カスタマーサービスに電話することなく注文を変更・キャンセルできる機能——を提供する。

**背景・コンテキスト**: 顧客満足度調査は、注文追跡とコミュニケーションを最大のペインポイントとして特定している。現在、注文ステータスは 4 時間ごとのバッチ更新で提供されており、システムが情報を持っているにもかかわらず顧客に届いていないため、顧客がサポートに状況を問い合わせる電話をかけている。セルフサービスの注文キャンセルは提供されておらず、顧客は電話する必要があり、レガシーシステムでの手動キャンセルプロセスの複雑さから、その 34% がエスカレーションに至っている。

**ドライバー強度**: 高

**推進要因**:
- セルフサービス顧客ポータルを支えるリアルタイム注文ステータス API
- 注文ライフサイクルの各遷移でのイベント駆動型通知（メール / SMS / Webhook）
- 対象注文のセルフサービスによるキャンセルと変更

**阻害要因**:
- 顧客向け機能は受注処理コアの先行移行に依存しており、ほとんどの移行ロードマップでは顧客体験の改善はフェーズ 2 以降の成果物

**関連ステークホルダー**: VP コマーシャル（NPS への影響）、カスタマーサービス（コール量の削減）、VP オペレーション（フルフィルメント精度の上流）

---

### SD-12: 物流 / フルフィルメントパートナー — 連携の信頼性

**ステークホルダー**: サードパーティ物流・フルフィルメントパートナー (3PL)

**ドライバー区分**: 運用

**ドライバーステートメント**: 脆弱なバッチ型ファイル連携を信頼性の高いリアルタイム API 連携に置き換える。これにより、連携障害を削減し、注文出荷を迅速化し、レガシーの専用連携作業を待つことなくパートナーが新機能をオンボードできるようにする。

**背景・コンテキスト**: 物流パートナーとの現在の連携は、夜間バッチスケジュールによる SFTP ファイル交換を通じて行われている。連携障害は月平均 3 件発生し、双方の手動対応を要する。標準 API レイヤーが存在しないため、新しい物流パートナーの追加には専用の連携作業で 3〜4 ヶ月を要する。あるパートナーは、連携障害によって生じた SLA 違反に対してペナルティ費用を請求することを示唆している。

**ドライバー強度**: 中

**推進要因**:
- バージョン管理された文書化された REST またはイベントストリーミング API レイヤー（ARC-000-PRIN-v1.0 の原則 13）
- セルフサービスドキュメントを通じて新パートナーがオンボードできる標準化された連携パターン
- バッチファイルに代わるリアルタイム注文出荷イベント

**阻害要因**:
- パートナーが新 API を利用するために自社の連携コードを更新する必要があり、調整とリードタイムが必要
- 適切なバージョニングと廃止通知なしの API 変更がパートナー連携を破壊する

**関連ステークホルダー**: VP コマーシャル（パートナー関係）、エンジニアリング（API 設計）、CISO（パートナー認証とアクセス制御）

### ゴール G-1: コア受注処理のクラウドネイティブプラットフォームへの移行

**由来するドライバー**: SD-1、SD-3、SD-4、SD-10

**ゴールオーナー**: CTO / CIO

**ゴールステートメント**: すべてのコア注文作成・管理・フルフィルメント機能のクラウドネイティブプラットフォームへの段階的移行を Month 18 までに完了する。注文データの損失をゼロとし、移行中もいかなる月においても現行の 99.0% 稼働率ベースラインを下回らない。

**重要性の理由**: 新プラットフォームは他のすべてのゴールを実現するための前提条件である。これなしにはコスト削減、コンプライアンス改善、顧客体験向上は実現できない。段階的アプローチ（ARC-000-PRIN-v1.0 の原則 1 に整合）は、CTO の近代化使命と VP オペレーションのリスク制約の双方を満たす。

**成功指標**:
- **主要指標**: 新プラットフォームで処理される注文量の割合（目標: Month 18 までに 100%）
- **副次指標**:
  - フェーズごとに廃止されたレガシーコンポーネントの数
  - フェーズごとの新プラットフォーム稼働率（目標: フェーズ 2 以降 99.5% 以上）
  - 照合レポートで確認されたゼロ注文データ損失イベント

**ベースライン**: 新プラットフォーム上の注文量 0%; レガシーシステムの稼働率 99.0%（直近 12 ヶ月）

**目標**: Month 18 までに新プラットフォームで注文量 100%; 新プラットフォーム稼働率 99.5% 以上

**測定方法**: API ゲートウェイの注文ルーティングメトリクス; フェーズごとのレガシーと新プラットフォームの注文数を比較する自動照合レポート; 稼働率モニタリングダッシュボード

**依存関係**:
- フェーズ 1 実装開始前にアンチコラプションレイヤーの設計承認を取得
- 移行フェーズ中、競合する優先事項からエンジニアリングチームの容量を保護
- 各フェーズゲートで UAT と移行サインオフのために運用チームの参加を確保

**達成リスク**:
- 二重稼働期間中のレガシーシステム不安定性が移行の緊急性を高め、安定化に使える時間を削減
- レガシーデータモデルの予想以上の複雑さがフェーズ 1 を遅延させ、後続フェーズを圧縮する

### ゴール G-2: 総所有コストの 30% 削減

**由来するドライバー**: SD-2、SD-8、SD-9

**ゴールオーナー**: CFO / 財務責任者

**ゴールステートメント**: ライセンス費、サポート契約、外部委託コスト、手動照合作業を含む Year 0 TCO ベースラインと比較して、プログラム開始から 24 ヶ月以内に受注管理プラットフォームの年間 TCO を 30% 削減する。

**重要性の理由**: CFO は業務効率化目標について取締役会にコミットメントしている。TCO 削減の実証は投資ケースを検証し、年次予算サイクルを通じてプログラムへの継続的な資金提供を確保する。

**成功指標**:
- **主要指標**: 年間 TCO（Year 2 対 Year 0 ベースライン）——目標: 30% 削減
- **副次指標**:
  - 排除されたレガシーライセンスおよびサポート契約コスト（目標: Month 18 までに 100% 排除）
  - 月次手動照合イベント数（現状: 1,200 件; 目標: Month 24 までに 100 件未満）
  - 照合作業に費やす財務 FTE 時間（現状: 約 2 FTE; 目標: 0.25 FTE 未満）

**ベースライン**: Year 0 TCO はプログラム開始時に財務と正式に確認・合意（推定: レガシーライセンス + サポート + 外部委託 + 照合作業 = 確認予定）

**目標**: Year 0 ベースライン比 30% TCO 削減、Month 24 の財務レビューで確認

**測定方法**: TCO ベースラインモデルに対するプログラム財務追跡; 照合作業の給与データ; 新プラットフォーム支出のクラウドコスト管理ツール

**依存関係**:
- レガシー廃止マイルストーンをスケジュール通りに達成（レガシーが並行稼働している間はコストが継続）
- プログラム開始時の TCO ベースライン化作業への財務チームの参画

**達成リスク**:
- 延長された二重稼働期間がレガシーコスト排除前にクラウド支出を増加させる
- 隣接システム（ERP、CRM）へのスコープ拡大が、受注管理ドメインでの比例する TCO メリットなしにプログラムコストを増加させる

### ゴール G-3: 受注処理稼働率 99.9% とエラー率 80% 削減

**由来するドライバー**: SD-4、SD-5、SD-11

**ゴールオーナー**: VP オペレーション

**ゴールステートメント**: Month 12（本番稼働後の最初の安定フェーズ）までに新プラットフォームで受注処理の稼働率 99.9% を達成し、Month 18 までに直近 12 ヶ月ベースラインと比較して注文エラー率（誤ったステータス、重複注文、注文消失）を 80% 削減する。

**重要性の理由**: 運用の信頼性は VP オペレーションの主要 KPI であり、VP コマーシャルの顧客満足度目標の前提条件である。このゴールの達成失敗は、オペレーションの観点から近代化全体のビジネスケースを損なう。

**成功指標**:
- **主要指標**: 新プラットフォームの月次稼働率（目標: Month 12 以降 99.9% 以上）
- **副次指標**:
  - インシデントからの平均復旧時間 (MTTR)（現状: 約 3 時間; 目標: 30 分未満）
  - 注文 10,000 件あたりの注文エラーイベント（現状ベースラインは Month 1 に計測; 目標: 80% 削減）
  - 月次の手動復旧手順の発動件数（現状: 14 件; 目標: 3 件未満）

**ベースライン**: レガシーシステム 12 ヶ月稼働率: 99.0%; 注文エラー率と MTTR は Month 1 に正式計測

**目標**: 稼働率 99.9%、エラー率 80% 削減、MTTR 30 分未満——すべて Month 18 までに達成

**測定方法**: アプリケーションパフォーマンス監視; 自動アラートとインシデント追跡; 注文照合レポート

**依存関係**:
- フェーズ 1 から自動ヘルスチェックとアラートを完全計装（ARC-000-PRIN-v1.0 の原則 6）
- 各フェーズが完全本番稼働する前にカオス・フォールトインジェクションテストを完了
- 新プラットフォームに本番トラフィックをルーティングする前に、オンコールランブックを作成・演習

**達成リスク**:
- 初期フェーズでのオブザーバビリティ計装が不十分でインシデントの検出が遅れ、MTTR が増大する
- 移行ウィンドウ中にレガシーシステムがさらに劣化し、オペレーションチームの注意が新プラットフォーム安定化から離れる

### ゴール G-4: 注文・出荷時間の短縮とリアルタイム顧客追跡の実現

**由来するドライバー**: SD-5、SD-11、SD-12

**ゴールオーナー**: VP コマーシャル / 営業

**ゴールステートメント**: Month 18 までに標準注文の平均 order-to-ship 時間を 48 時間から 24 時間に短縮し、Month 12 までに顧客にリアルタイムの注文ステータス（ステータス変更後 5 分以内の更新、現行の 4 時間バッチサイクルの置き換え）を提供する。

**重要性の理由**: リアルタイム追跡は最大の顧客不満を直接解消し NPS を改善する。order-to-ship 時間の短縮は、商業チームがパイプラインに持つ新しい商業 SLA ティアの実現要件である。

**成功指標**:
- **主要指標**: 平均 order-to-ship 時間（目標: 標準注文で 24 時間）
- **副次指標**:
  - 注文ステータス更新レイテンシ（現状: 最大 4 時間; 目標: 5 分未満）
  - 注文ステータス問い合わせに関するカスタマーサービスのコール量（現状ベースラインは計測予定; 目標: 30% 削減）

**ベースライン**: 平均 order-to-ship: 48 時間; ステータス更新レイテンシ: 最大 4 時間; NPS はベースライン化予定

**目標**: 24 時間 order-to-ship; 5 分未満のステータス更新; Month 18 までに NPS +15

**測定方法**: 受注管理アナリティクス; 顧客満足度調査（四半期ごと); 注文ステータスコール量に関するコールセンターレポーティング

### ゴール G-5: セキュリティ・コンプライアンスポスチャの達成と維持

**由来するドライバー**: SD-6、SD-7

**ゴールオーナー**: CISO / コンプライアンス責任者

**ゴールステートメント**: 12 ヶ月のプログラムウィンドウ内にレガシープラットフォームの深刻度「高」3 件・「中」11 件すべての所見を是正する。8 ヶ月以内に新プラットフォームで PCI-DSS 再認証を達成する。Month 12 までに GDPR 準拠のデータライフサイクル管理（自動化された消去の権利を含む）を実装する。

**重要性の理由**: 深刻度「高」の 2 件はレガシープラットフォームから移行することなしに是正できず、移行とセキュリティ是正は同一のアクションである。PCI-DSS 再認証には動かせない外部期限がある。

**成功指標**:
- **主要指標**: 未解決の深刻度「高」セキュリティ所見（現状: 3 件; 目標: Month 8 までにゼロ）
- **副次指標**:
  - Month 8 までの PCI-DSS 再認証達成（合否）
  - GDPR 個人情報開示請求 (SAR) の対応時間（現状: 47 日; 目標: Month 12 までに 5 日未満）
  - GDPR 消去の権利の自動化カバレッジ（目標: Month 12 までにすべての個人データフィールドを 100% 自動化）

**ベースライン**: 深刻度「高」3 件・「中」11 件; PCI-DSS 再認証期限まで 8 ヶ月; SAR 対応時間: 47 日

**目標**: Month 8 までに深刻度「高」所見ゼロ; PCI-DSS 再認証済み; SAR 対応 5 日未満

**測定方法**: セキュリティ所見トラッカー（ペネトレーションテストごとに更新）; PCI-DSS QSA レポート; SAR 対応時間ログ; 自動データライフサイクル監査レポート

**依存関係**:
- 移行のフェーズ 1 に PCI-DSS スコープ内のコンポーネントを含め、新プラットフォームでの再認証を可能にする
- データモデル設計が確定する前の Month 1 にデータ分類演習（ARC-000-PRIN-v1.0 の原則 9）を完了

**達成リスク**:
- PCI-DSS スコープコンポーネントが後の移行フェーズにあり、それらが移行されるまで新プラットフォームでの認証達成が不可能
- レガシーデータモデルに未発見の個人情報フィールドがあり、消去の権利の自動化が予想より困難

### ゴール G-6: フィーチャー提供リードタイムの 75% 削減

**由来するドライバー**: SD-1、SD-3、SD-10

**ゴールオーナー**: CTO / CIO

**ゴールステートメント**: Month 18 までに受注管理フィーチャーのフィーチャー要求から本番デプロイまでの平均所要時間を 8 週間から 2 週間に短縮し、レガシーベースラインと比較して本番インシデントを引き起こすデプロイを 60% 削減する。

**重要性の理由**: デリバリーベロシティは CTO のアーキテクチャ健全性の主要指標であり、商業チームが市場機会に対応する能力に直結している。現在の 8 週間サイクルは、商業チームが受注管理への変更要求を停止させ、代わりにコストのかかる回避策に要求をルーティングする原因となっている。

**成功指標**:
- **主要指標**: 受注管理フィーチャー提供の平均リードタイム（現状: 8 週間; 目標: 2 週間）
- **副次指標**:
  - デプロイ頻度（現状: 隔週; 目標: 毎日またはオンデマンド）
  - 変更失敗率（デプロイが引き起こす本番インシデント; 現状: 約 35%; 目標: 10% 未満）
  - デプロイ起因インシデントからの平均復旧時間（現状: 3 時間; 目標: 30 分未満）

**ベースライン**: 8 週間のフィーチャーリードタイム; 隔週デプロイ; 35% の変更失敗率

**目標**: 2 週間リードタイム; 毎日デプロイ; 10% 未満の変更失敗率——Month 18 までに達成

**測定方法**: CI/CD パイプラインメトリクス（デプロイ頻度、リードタイム); インシデント追跡システム（デプロイ相関); チームベロシティ追跡

**依存関係**:
- フェーズ 1 から自動テストとロールバックを備えた CI/CD パイプラインの稼働
- フィーチャーの提供を求められる前に新スタックで完全にトレーニングを受けたエンジニア
- 調整オーバーヘッドなくチームが独立してデプロイできるよう定義されたアーキテクチャ境界

**達成リスク**:
- レガシー保守と新規開発に分断されたエンジニアチームが両方のベロシティを低下させる
- 自動テストカバレッジが不十分で、エンジニアが継続的デプロイに自信を持てない

### 対立分析

**対立 C-1: CFO のコスト実現の急進性と VP オペレーションの移行リスク回避**

CFO はコスト節約をできるだけ早く実現するために移行フェーズを迅速に進めたい（二重稼働期間はコストの重荷）。VP オペレーションは、新プラットフォームの信頼性を確認してから負荷を増やすために、各フェーズに延長された安定化期間を設けることを望む。

- **解決戦略**: フェーズごとに時間制限付きの安定化期間（例: 4 週間の並行稼働、終了基準はエラー率 5% 未満）を定義し、オープンエンドの安定化ではなく明確な安全策を与える。これにより財務には予測可能な移行スケジュール、オペレーションには明確な安全策を提供する。フェーズゲート基準は Month 1 に合意し、プログラム途中で再交渉しない。

**対立 C-2: レガシー保守と新規ビルドの間でのエンジニアリングチームの分断**

CTO は新プラットフォームに最大のエンジニアリング容量を割り当てたい。VP オペレーションは移行中もレガシーシステムを安定させ続けるために継続的なエンジニアリングサポートを必要とする（これらは同一の有限なエンジニアリングリソースを奪い合う）。

- **解決戦略**: プログラム開始前に専任のレガシー保守スクワッド（最小実行可能チーム）を定義し、リングフェンスする。モダンプラットフォームチームは別途リソースを確保し、移行フェーズ中はレガシー保守にローテーションしない。

**対立 C-3: PCI-DSS 再認証期限と移行フェージング**

コンプライアンス期限（8 ヶ月）により、技術的な準備状況に関わらず PCI-DSS スコープのコンポーネントが移行のフェーズ 1 に強制される可能性があり、それらのコンポーネントのアンチコラプションレイヤーの設計とテスト時間が圧縮される。

- **解決戦略**: ARC-000-PRIN-v1.0 の原則 9 に基づくデータ分類演習の一環として、PCI-DSS スコープコンポーネントを Month 1 に特定する。それらのコンポーネントをフェーズ 1 で安全に移行できない場合、移行が進む間レガシープラットフォームでの認証を維持するための補完的コントロールを QSA と交渉する——これは最初の 1 ヶ月以内に CISO とコンプライアンス責任者にエスカレーションする。

### 相乗効果

**相乗効果 S-1: SD-4（運用の信頼性）と SD-11（顧客体験）** — 両者は同一のゴール（G-3: 稼働率 99.9%、エラー率 80% 削減）によって満たされる。信頼性への投資は、社内の運用チームとエンドカスタマーの双方に同時にメリットをもたらし、このゴールにプログラム内で最も幅広いステークホルダーの支持を与える。

**相乗効果 S-2: SD-6（セキュリティ是正）と SD-7（GDPR コンプライアンス）** — セキュリティとコンプライアンスのゴール（G-5）は共同オーナーシップで、同じ証跡ベース（監査ログ、データ分類）を共有する。単一の不変監査証跡とデータ分類演習が両ステークホルダーを満たし、重複した作業を削減する。

**相乗効果 S-3: SD-2（コスト削減）と SD-8（照合の排除）** — 財務チームの照合の痛み（SD-8）は運用コストでも財務コストでもある。これを排除することで財務チームの運用ドライバーが満たされ、同時に CFO の TCO 30% 目標（G-2）に直接貢献する。SD-12（物流パートナーのバッチ処理）を修正するのと同じイベント駆動型連携が ERP 照合問題も解決し、連携近代化への投資が 3 つのステークホルダーグループを同時に対応する。

**相乗効果 S-4: SD-3（エンジニアリング近代化）と SD-10（開発者体験）** — CTO とエンジニアリングは完全に整合したドライバーを持つ。CI/CD パイプライン、自動テスト、モダンツールへの投資（ARC-000-PRIN-v1.0 の原則 19〜21）はトレードオフなく両者を同時に満たす。

## コミュニケーション・エンゲージメント計画

### エグゼクティブスポンサー (CEO/COO)

**主要メッセージ**: プログラムは、各フェーズゲートでビジネス価値を提供するリスク管理された段階的アプローチを通じて、24 ヶ月以内に競合上の能力格差を埋めるトラックにある。

**重要な話題**:
- 明確なゴー/ノーゴー基準を持つフェーズごとのマイルストーンがビジネス継続性を守る
- 初期フェーズでリアルタイム追跡と API 機能が実現され、商業パイプラインがアンブロックされる
- ARC-000-PRIN-v1.0 に確立されたアーキテクチャ原則が、官僚的な遅延を生まずにすべての決定を管理する

**コミュニケーション頻度**: 隔週ステアリングボード; 必要に応じてエスカレーション

**優先チャネル**: ステアリングボード会議; エグゼクティブダッシュボード（稼働率、移行 %、コスト傾向）

**成功事例**: 6 ヶ月時点——最初のフェーズが稼働、リアルタイム注文追跡が開始、移行中の計画外障害ゼロ

---

### CFO / 財務責任者

**主要メッセージ**: 投資は Month 24 までに TCO 30% 節約を実現するスケジュール通りに進んでおり、早期コスト指標（外部委託費削減、部分的なレガシー廃止）が Month 9 までに確認できる。

**重要な話題**:
- 二重稼働期間のコストをモデル化し予測に対して追跡
- 移行フェーズに連動した照合イベント数の削減
- 合意されたベースラインに対して四半期ごとに確認されるビジネスケース ROI

**コミュニケーション頻度**: 月次財務レビュー; 四半期ビジネスケース更新

**優先チャネル**: 財務ダッシュボード; 月次書面レポート

**成功事例**: 12 ヶ月時点——最初のレガシーコンポーネントが廃止、ライセンス請求行が消滅、照合イベントが 40% 削減

---

### CTO / CIO

**主要メッセージ**: アーキテクチャは ARC-000-PRIN-v1.0 の原則——モジュラー、イベント駆動型、クラウドネイティブ——に基づいて構築されており、エンジニアリングチームがこのプログラムを超えてデリバリーを加速するモダンスキルを習得している。

**重要な話題**:
- ADR がすべての重要な決定を文書化し、アーキテクチャの根拠が保存される
- CI/CD パイプラインとテスト自動化がフィーチャーデリバリー前のフェーズ 1 で構築される
- エンジニアの流出リスクをテクニカルメトリクスと並ぶ先行指標として追跡

**コミュニケーション頻度**: 週次プログラム同期; フェーズゲートでのアーキテクチャレビュー

**優先チャネル**: アーキテクチャレビューセッション; プログラム Slack / コラボレーションツール

**成功事例**: 9 ヶ月時点——エンジニアリングチームが変更失敗率 10% 未満で毎日本番デプロイ; モダンスタックを入社理由として挙げる最初のシニアエンジニア採用

---

### VP オペレーション

**主要メッセージ**: すべての移行フェーズは本番稼働前にテスト済みの明示的なロールバック計画、定義された終了基準を持つ安定化ウィンドウ、サインオフへのオペレーションの完全な参画を持つ。

**重要な話題**:
- オペレーションチームメンバーはフェーズ計画への指名された参加者であり、単なる承認者ではない
- 新プラットフォームのオブザーバビリティがレガシーシステムよりも優れたリアルタイム可視性をオペレーションに提供する
- 二重稼働の終了基準は事前に合意——オペレーションは早期切替を強制されない

**コミュニケーション頻度**: 週次運用同期; フェーズ切替ウィンドウ中は毎日

**優先チャネル**: 運用ランブックレビュー; モニタリングダッシュボード; プログラムマネージャーへの直接チャネル

**成功事例**: 12 ヶ月時点——3 ヶ月連続で稼働率 99.9% を確認; MTTR が 3 時間から 25 分に削減

---

### VP コマーシャル / 営業

**主要メッセージ**: リアルタイム注文追跡と B2B 顧客ポータルを実現する API レイヤーはフェーズ 2 の成果物——商業パイプラインは定義されたスケジュールでアンブロックされる。

**重要な話題**:
- フェーズ 2 マイルストーン（Month 12）がセルフサービスポータルの API 基盤を提供
- order-to-ship 短縮と NPS 改善が四半期ごとに計測・報告される
- 新商業モデル（サブスクリプション、SLA ティア）がアーキテクチャ的にフェーズ 3 から実現可能

**コミュニケーション頻度**: 月次商業影響ブリーフィング; 各フェーズ前のリリースプレビュー

**優先チャネル**: 商業ダッシュボード; プレリリースブリーフィングセッション

**成功事例**: 12 ヶ月時点——リアルタイム追跡が稼働、カスタマーサービスの注文ステータスコール量が 25% 削減、B2B ポータルが開発中

---

### CISO / コンプライアンス責任者

**主要メッセージ**: セキュリティコントロールはフェーズ 1 に組み込まれており——後付けではない。PCI-DSS スコープコンポーネントは再認証期限に合わせて移行シーケンスで優先される。

**重要な話題**:
- 不変監査証跡とシークレット管理はフェーズ 1 の成果物であり、先送りではない
- PCI-DSS スコープコンポーネントは Month 1 に特定され、必要に応じて移行シーケンスを調整
- セキュリティアーキテクチャは本番稼働承認前の各フェーズゲートでレビューされる

**コミュニケーション頻度**: 月次セキュリティレビュー; フェーズゲートサインオフ

**優先チャネル**: セキュリティ設計レビューセッション; 所見トラッカー; コンプライアンス証跡パック

**成功事例**: 8 ヶ月時点——新プラットフォームで PCI-DSS 再認証達成; 深刻度「高」の未解決所見ゼロ

---

### 受注フルフィルメントチーム

**主要メッセージ**: 既存のワークアラウンドに関するあなたたちの運用知識が新システム設計に組み込まれます——この移行はワークアラウンドを新技術に移植するだけでなく、排除します。

**重要な話題**:
- フルフィルメントチームメンバーは、ワークフローに影響するすべてのフェーズの UAT に参加する
- トレーニングは切替の 4 週間前（前日ではなく）に提供される
- 各フェーズ切替後 30 日間はフィードバックチャネルが開いている

**コミュニケーション頻度**: スプリントレビュー; UAT フェーズ中は週次; 切替ウィンドウ中は毎日

**優先チャネル**: 直接チームセッション; UAT フィードバックフォーム; チームチャットチャネル

**成功事例**: 6 ヶ月時点——最初のフェーズが稼働、14 件の文書化されたワークアラウンドのうち 3 件が排除、スタッフオンボーディング時間が削減

**抵抗層**（懐疑的または反対）:
- **受注フルフィルメントチーム** — 新しい包装で同じ問題が再現されると約束された改善の歴史; 完成した決定のサインオフではなく、設計に参画させることで信頼を築かなければならない。戦略: Month 1 から設計ワーキンググループに指名されたチームメンバーを配置し、サインオフ前に各ワークアラウンドを（再現するのではなく）排除するという文書化されたコミットメントを行う。

## リスク登録簿（ステークホルダー関連）

### R-1: オペレーションの抵抗が移行フェーズを遅延させる

**関連ステークホルダー**: VP オペレーション、受注フルフィルメントチーム

**リスク説明**: VP オペレーションが、不十分な安定化時間または未解決の運用懸念を理由にフェーズサインオフを差し控え、二重稼働期間を予測を超えて延長し、プログラムコストを増加させる。

**ゴールへの影響**: G-1（移行タイムライン）、G-2（TCO 削減の遅延）、O-1、O-2

**発生確率**: 中

**影響**: 高

**緩和戦略**: 移行が始まる前の Month 1 にフェーズ終了基準（稼働率閾値、エラー率目標）を合意する。オペレーションはこれらの基準をレビューし、サインオフする——基準は裁量的な判断ではなく客観的なゲートとなる。最初から移行設計にオペレーションを含める。

**コンティンジェンシープラン**: エグゼクティブスポンサーによるステアリングボード裁定; ゲート紛争を解決するための時間制限付きエスカレーションプロセス（最大 5 営業日）。

---

### R-2: PCI-DSS 期限がセキュリティクリティカルな移行フェーズを圧縮する

**関連ステークホルダー**: CISO、コンプライアンス責任者

**リスク説明**: PCI-DSS スコープコンポーネントが Month 8 までに移行の準備ができておらず、レガシープラットフォームでの部分的な認証か、セキュリティ設計レビューをバイパスする圧縮された移行のどちらかを強いられる。

**ゴールへの影響**: G-5（コンプライアンス）、O-4

**発生確率**: 中

**影響**: 高

**緩和戦略**: Month 1 に PCI-DSS スコープコンポーネントを特定する。Month 7 までに（認証に 1 ヶ月を確保するため）安全に移行が完了できない場合、QSA に直ちにコンタクトして代替計画を合意する——レガシープラットフォームでの補完的コントロール延長か、スコープ内コンポーネントの並行ワークストリームとしての加速移行。

**コンティンジェンシープラン**: 文書化された移行コミットメントレターを伴う QSA 合意済み暫定認証; 補完的コントロールが不十分な場合は CISO がエグゼクティブスポンサーにエスカレーション。

### R-3: 物流パートナーの採用がリアルタイム連携を遅延させる

**関連ステークホルダー**: 物流 / フルフィルメントパートナー、VP コマーシャル

**リスク説明**: 1 社以上の主要物流パートナーが SFTP バッチから新 API への連携移行に時間がかかり、リアルタイム注文出荷と NPS 改善が Month 12 を超えて遅延する。

**ゴールへの影響**: G-4（order-to-ship、リアルタイム追跡）、O-3

**発生確率**: 中

**影響**: 中

**緩和戦略**: Month 2 に API プレビュードキュメントと移行タイムラインで物流パートナーとエンゲージする。可能であれば連携移行を契約上のマイルストーンとして含める。移行期間中にパートナーの SFTP ファイルを受け取り内部イベントに変換する互換性アダプターを構築し、パートナー依存をクリティカルパスから除く。

**コンティンジェンシープラン**: すべてのパートナーが移行するまで SFTP アダプターを本番で維持する; パートナーへの書面通知を伴い SFTP サポートを撤廃する内部期限（Month 18）を設定する。

### R-4: エンジニアリング容量がレガシー保守に消費される

**関連ステークホルダー**: エンジニアリングチーム、CTO、VP オペレーション

**リスク説明**: 継続的なレガシーシステムのインシデントと保守要求が、新プラットフォーム開発に必要なエンジニアリング容量を消費し、移行フェーズの遅延を引き起こす。

**ゴールへの影響**: G-1（移行タイムライン）、G-6（デリバリーベロシティ）

**発生確率**: 高

**影響**: 高

**緩和戦略**: プログラム開始前に専任のレガシー保守スクワッド（最低 2 名のエンジニア）を定義・資金確保する。モダンプラットフォーム作業からリングフェンスする。レガシー変更の正式なリクエストプロセスを高い基準で確立する——移行中はセキュリティと緊急本番修正のみを受け付ける。

**コンティンジェンシープラン**: レガシーインシデントが保守スクワッドの容量を超えてエスカレートした場合、エグゼクティブスポンサーがモダンプラットフォームチームから引き抜くのではなく、一時的な外部委託サポートを承認する。

### R-5: 受注フルフィルメントチームの UAT からの離脱

**関連ステークホルダー**: 受注フルフィルメントチーム、VP オペレーション

**リスク説明**: 受注フルフィルメントスタッフが日常業務で手一杯で、UAT に実質的に参加できず、未発見のユーザビリティ問題が本番稼働して信頼を損なう。

**ゴールへの影響**: G-3（エラー率削減）、O-3

**発生確率**: 中

**影響**: 中

**緩和戦略**: UAT 期間は VP オペレーションと事前に合意する; UAT 参加者のバックフィルまたは作業負荷軽減を手配する。UAT はオープンエンドなレビューではなく開始日と終了日を持つ指名された成果物。チームリードはプログラムマネージャーに直接ブロッカーを提起する権限を持つ。

**コンティンジェンシープラン**: UAT 容量が不十分な場合、UAT スコープを削減するのではなくフェーズ本番稼働日を延長する。品質は妥協しない。
