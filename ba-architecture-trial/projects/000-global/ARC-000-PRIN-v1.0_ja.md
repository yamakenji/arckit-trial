## エグゼクティブサマリー

本文書は、レガシー受注管理システム モダナイゼーション プログラムにおけるすべての技術的意思決定を規定するアーキテクチャ原則を定めるものである。これらの原則は、レガシーシステムからクラウドネイティブアーキテクチャへの移行を通じて、一貫性・セキュリティ・スケーラビリティ・事業戦略との整合を確保するために策定された。

**適用範囲**: 受注管理モダナイゼーション プログラム内のすべての技術ワークストリーム、および依存・連携する関連システム

**権限機関**: エンタープライズアーキテクチャレビュー委員会 (ARB)

**コンプライアンス**: 第VI章に定める手続きを経て適用除外が承認された場合を除き、すべてのチームに対して遵守義務がある

**設計思想**: 本原則は**技術スタック非依存**である。すなわち、アーキテクチャが備えるべき「品質」を規定するものであり、特定製品による実装方法を規定するものではない。クラウドサービスや製品の選定は、本原則に準拠した上で、リサーチフェーズおよび設計フェーズにて実施する。

## I. 戦略的原則

### 1. 段階的モダナイゼーション（一括移行の禁止）

**原則ステートメント**:
レガシー受注管理システムは、実績のある移行パターン（ストラングラーフィグ (Strangler-Fig) や抽象化によるブランチ分離など）を用いて段階的にモダナイズしなければならない。各フェーズにおいて業務継続性を維持することが必須条件であり、一括での全面置き換えは禁止する。

**根拠**:
一括移行は、複雑な受注管理システムにおいて歴史的に成功率が低い。段階的デリバリーにより、リスクを管理しながら早期に価値を提供でき、次のフェーズに移行する前に学びを取り込むことができる。

**示唆**:

- レガシーコンポーネントとモダンコンポーネントが本番環境で共存できる明確な機能的境界を定義する
- 各移行フェーズ完了後、システムは完全に運用可能な本番デプロイ可能な状態でなければならない
- 並行稼働期間はあらかじめ計画し、期間を明示し、終了条件を定義する
- 腐敗防止層 (Anti-Corruption Layer) によってモダンドメインモデルをレガシーデータ構造から隔離する
- 移行フェーズは業務価値と技術的リスクを基準に優先順位を付ける

**検証ゲート**:

- [ ] 各移行フェーズの明確な入口・出口条件が文書化されている
- [ ] 腐敗防止層の設計がレガシーとの統合前にレビューされている
- [ ] 並行稼働期間が日付と運用指標によって限定されている
- [ ] 各移行フェーズのロールバック計画が定義され、テストされている
- [ ] どのフェーズも、そのフェーズ以前には存在しなかった単一障害点を導入していない

---

### 2. クラウドネイティブ設計

**原則ステートメント**:
新規および移行済みのすべてのコンポーネントは、クラウドネイティブ機能（マネージドサービス・弾力的スケーリング・従量課金・インフラ自動化）を最大限に活用するよう設計しなければならない。リアーキテクチャを伴わないリフト・アンド・シフトは、再アーキテクチャ実施予定日が定められている過渡的状態においてのみ許容する。

**根拠**:
クラウドネイティブ設計により、弾力的なスケーラビリティ・運用効率・差別化されない運用負荷の削減が実現できる。リアーキテクチャを伴わないリフト・アンド・シフトは技術的負債を先送りし、クラウド移行の主要な商業上のメリットを享受できない。

**示唆**:

- 要件を満たすマネージドプラットフォームサービスを、自己管理型の同等サービスより優先する
- 業務ロジックが許す限り、ステートレスなコンポーネントを設計する
- エフェメラルで不変なインフラを前提とした設計を採用する（手動パッチや in-place アップグレードを排除）
- リフト・アンド・シフトコンポーネントには、目標日付を記載した追跡済みのリアーキテクチャ バックログ項目を設ける

**検証ゲート**:

- [ ] 各ワークロードの設計前にクラウドネイティブ適合性を評価している
- [ ] リフト・アンド・シフトコンポーネントにはリアーキテクチャ目標日が文書化されている
- [ ] ステートフルなコンポーネントが識別され、必要性が正当化されている
- [ ] コストモデルが可変容量の料金体系を反映している

---

### 3. スケーラビリティと弾力性

**原則ステートメント**:
すべてのシステムは、負荷に応じて水平スケーリングにより需要を満たし、動的に容量を調整できるよう設計しなければならない。ピーク時の注文処理スループットを制限する単独のボトルネックとなるコンポーネントがあってはならない。

**根拠**:
注文量は季節性・イベント性（販促・キャンペーン・期末急増など）を持つ。アーキテクチャは、手動介入や平時に遊休するプロビジョニング済みのヘッドルームなしに、注文スパイクを処理できなければならない。

**示唆**:

- 水平方向にレプリケートできるステートレスコンポーネントを設計する
- ハードコードされた上限や固定容量の前提を排除する
- すべてのコンピュートレイヤーでロードバランシングを必須とする
- 本番環境への適用前に、オートスケーリングポリシーを定義してテストする
- 各コンポーネントのスループット上限を文書化し、ピーク予測と照合してレビューする

**検証ゲート**:

- [ ] リアーキテクチャなしに増加した負荷の下で水平スケーリングできる
- [ ] 水平スケーリングを妨げる単独のボトルネックが存在しない
- [ ] ピーク注文量の2倍でのロードテストがパフォーマンス目標を達成している
- [ ] オートスケーリングのメトリクスと閾値が定義されている
- [ ] コストモデルが可変容量を考慮している

---

### 4. レジリエンスと耐障害性

**原則ステートメント**:
すべてのシステムは、依存関係に障害が発生した際もグレースフルデグラデーション（段階的縮退）を行い、データ損失や手動介入なしに自動的に回復しなければならない。ダウンストリームシステムが利用不能な場合においても、注文処理の整合性を保持しなければならない。

**根拠**:
受注管理システムは、決済ゲートウェイ・在庫システム・物流プロバイダー・顧客通知サービスと連携しており、これらはいずれも一時的に利用不能になる可能性がある。アーキテクチャはこれらの障害発生を前提とし、注文が無通知のまま消失しないことを保証しなければならない。

**示唆**:

- すべての外部・内部サービス依存に対してサーキットブレーカーを実装する
- すべてのネットワーク呼び出しにタイムアウトを設定し、フォールバック動作を文書化する
- 一時的な障害には指数バックオフ (Exponential Backoff) とジッターを用いたリトライを実装する
- ダウンストリームの処理をトリガーする前に、注文を永続的に保存しなければならない
- グレースフルデグラデーション: 通知サービスなど非クリティカルなダウンストリーム障害が注文受付をブロックしてはならない
- バルクヘッド分離によって注文処理フェーズ間のカスケード障害を防止する

**検証ゲート**:

- [ ] すべての外部統合における障害モードが識別され、対策が取られている
- [ ] 本番前にフォールトインジェクションまたはカオステストを実施している
- [ ] サービスレベルごとに目標復旧時間 (RTO) と目標復旧時点 (RPO) が定義されている
- [ ] 自動フェイルオーバーがエンドツーエンドでテストされている
- [ ] 部分的障害時の注文永続性が実証されている

### 5. セキュリティ・バイ・デザイン（非交渉事項 (NON-NEGOTIABLE)）

**原則ステートメント**:
すべてのアーキテクチャは、ゼロトラスト原則に基づく多層防御セキュリティを実装しなければならない。セキュリティ制御は基盤となる要件であり、実装後に追加するものではない。顧客の注文データと決済情報は攻撃価値の高い標的であり、相応に保護しなければならない。

**根拠**:
受注管理システムは、購買履歴・配送先住所などの機密性の高い顧客データを保有し、決済トークンを取り扱う可能性がある。情報漏洩はコンプライアンス上・財務上・評判上の重大な結果をもたらす。

**ゼロトラストの4原則**:

1. **IDベースのアクセス**: ネットワークベースの信頼を排除し、すべてのリクエストを認証・認可する
2. **最小権限**: 必要最小限の権限のみを付与し、サービスアカウントは単一の責務にスコープを絞る
3. **全経路での暗号化**: すべてのストアおよびキューにおいて、転送中・保存時のデータを暗号化する
4. **継続的な検証**: サービス間の呼び出しを含むすべてのアクセスパターンを監視・ログ記録・分析する

**必須制御**:

- [ ] 本番システムへのすべての人的アクセスに多要素認証 (MFA) を適用する
- [ ] サービス間認証には署名付きトークンまたは相互認証を使用する
- [ ] シークレットはシークレット管理サービスを通じて管理する。コード・設定ファイル・ログへの記載は禁止する
- [ ] 処理レイヤー間に最小信頼ゾーンによるネットワークセグメンテーションを実施する
- [ ] 注文データまたは顧客データを含むすべてのデータストアで保存時暗号化を適用する
- [ ] すべてのネットワーク通信に暗号化トランスポートを使用する（本番での平文プロトコルを禁止）
- [ ] すべての認証・認可・注文状態変更イベントの構造化監査ログを記録する
- [ ] 定期的な脆弱性スキャンおよびペネトレーションテストを実施する

**適用フレームワーク**:

- NIST Cybersecurity Framework (CSF 2.0)
- 決済カードデータがスコープ内の場合: PCI-DSS
- 顧客個人情報に関わるデータ保護法（GDPR または適用法令）

**適用除外**: なし。セキュリティ原則は非交渉事項である。特定の制御実装については、文書化された補償的制御を伴う場合に限り変更を許容する。

**検証ゲート**:

- [ ] 脅威モデリングが完了し、セキュリティチームによるレビューが行われている
- [ ] セキュリティ制御が要件にマッピングされている
- [ ] ソースコードにシークレットが含まれていないことを CI パイプラインで自動スキャンしている
- [ ] ペネトレーションテストのスケジュールが定義されている

### 6. オブザーバビリティと運用の卓越性

**原則ステートメント**:
すべてのシステムは、リアルタイム監視・注文ライフサイクルの可視性・トラブルシューティング・キャパシティプランニングを可能にする構造化されたテレメトリ（ログ・メトリクス・トレース）を出力しなければならない。注文の完全なライフサイクルが、受注から出荷完了までエンドツーエンドで追跡可能でなければならない。

**根拠**:
受注管理の運用は、注文ステータス・処理レイテンシー・エラー率・連携先の正常性に関するリアルタイムの可視性に依存する。観測できないものはサポートできない。

**テレメトリ要件**:

- **ログ**: 相関IDおよび注文参照IDをすべてのログ行に含む構造化ログ
- **メトリクス**: リクエスト量・処理レイテンシーパーセンタイル (p50, p95, p99)・エラー率・キュー深度
- **トレース**: すべての注文処理ホップを跨ぐ分散トレースコンテキストの伝播
- **アラート**: サービスレベル目標 (SLO) ベースのアラートと、文書化された実行可能なランブック

**必須の計装指標**:

- 注文受付率・処理レイテンシー・フェーズ別障害率
- 各ダウンストリームシステム（在庫・決済・物流）の連携正常性メトリクス
- 非同期処理のキュー深度とバックログ深度
- ビジネスメトリクス: 時間ウィンドウごとの注文作成・出荷完了・キャンセル・返金件数
- セキュリティイベント: 認証失敗・ポリシー違反・異常なアクセスパターン

**ログ保持期間**:

- **セキュリティ・監査ログ**: 最低1年間。法令要件については法務・コンプライアンスと協議する
- **アプリケーション・注文ログ**: 最低90日間
- **メトリクス**: トレンド分析のための適切な集約を伴い13ヶ月間

**検証ゲート**:

- [ ] すべてのコンポーネントにログ・メトリクス・トレースが計装されている
- [ ] 受注から出荷完了までのエンドツーエンドの注文トレースを実証できる
- [ ] 注文処理のSLOとSLIが定義されている
- [ ] 上位5件の運用障害シナリオのランブックが作成されている

## II. データ原則

### 7. 注文トランザクションの完全性

**原則ステートメント**:
注文の作成・更新・キャンセルを行うすべての操作は、アトミックかつ永続的でなければならない。注文を不定状態のままにする部分的な状態更新は禁止する。各注文の状態遷移は不変のイベントとして記録しなければならない。

**根拠**:
注文は財務的コミットメントを表す。部分的または不整合な注文状態（例: 決済は引き落とされたが注文が確定されない、または在庫は引き当て済みだが注文が消失するなど）は、顧客への直接的な被害・財務損失・調整コストを引き起こす。

**示唆**:

- 注文状態遷移は、障害時にロールバックするトランザクション書き込みを使用する
- データベース書き込みに加えて信頼性の高いイベント発行のためにアウトボックスパターン (Outbox Pattern) または同等の仕組みを使用する
- 重複処理を防ぐため、すべての注文作成・更新エンドポイントに冪等性キーを必須とする
- 複数システムにまたがる操作（例: 在庫引き当て → 決済請求 → 注文確定）にはサガパターン (Saga Pattern) を使用し、各ステップに補償トランザクションを定義する
- 複数のデータストアをまたがる分散トランザクションは禁止する

**検証ゲート**:

- [ ] 障害発生時に注文を不定状態のままにする注文状態遷移パスが存在しない
- [ ] すべての注文作成・変更操作について冪等性テストを実施している
- [ ] 各障害シナリオに対するサガの補償トランザクションがテストされている
- [ ] アウトボックスパターンまたは同等の仕組みが実装・テストされている
- [ ] サービス境界を跨ぐ分散トランザクションが存在しない

### 8. 不変の注文監査証跡

**原則ステートメント**:
注文へのすべての変更は、アクター識別子・タイムスタンプ・変更前の状態を含む不変のイベントとして記録しなければならない。注文の完全な履歴がいつでも照会可能でなければならない。

**根拠**:
注文の監査証跡は、顧客からの異議申し立て対応・財務調整・コンプライアンス対応・不正調査に必要である。記録は改ざん・削除されてはならない。

**示唆**:

- すべての注文状態変更に対してイベントソーシングまたは追記専用の監査ログを使用する
- 監査記録には、アクター識別子・タイムスタンプ・操作・変更前の状態・変更後の状態・ソースシステムを含める
- 監査記録は書き込み専用とし、監査データへの更新・削除操作は禁止する
- 保持期間は法務・コンプライアンス要件によって定義する（財務記録は最低7年間。適用法令を確認すること）
- 監査ログの完全性を改ざんから保護する（暗号的検証または同等の仕組み）

**検証ゲート**:

- [ ] 受注から現在の状態までの完全な注文履歴が照会可能である
- [ ] 監査記録を変更・削除するパスが存在しない
- [ ] 監査ログの保持期間が文書化され、自動化されている
- [ ] 監査ログのアクセス制御により、書き込みアクセスがシステムアカウントのみに制限されている

### 9. データ主権とデータ分類

**原則ステートメント**:
すべてのデータ資産は、ストレージ・処理・転送に関する設計上の意思決定を行う前に分類しなければならない。顧客の個人情報と注文データは、適用されるデータ保護法に準拠した管轄区域内に保存しなければならない。

**データ分類レベル**:

1. **公開 (Public)**: 制限なし（商品カタログ、公開価格など）
2. **内部 (Internal)**: 従業員アクセスのみ（運用メトリクス、内部レポートなど）
3. **機密 (Confidential)**: 必要最小限の知る必要に基づくアクセス（顧客の個人情報、注文履歴、配送先住所など）
4. **制限 (Restricted)**: 最高レベルの管理（決済トークン、認証情報など）

**データ所在地**:

- 顧客の個人情報および注文データは適用管轄の要件（GDPR または同等法令）に準拠しなければならない
- 国境を越えたデータ転送には、文書化された法的根拠が必要である
- クラウドリージョンの選定前にデータ所在地要件を確認しなければならない

**データ保持**:

- データクラスごとに定義された保持期間を超過したデータの自動削除を実施する
- 顧客の削除権（忘れられる権利）のプロセスを文書化し、実装可能にする（法的保全義務の適用を受ける場合を除く）
- 紛争・調査・行政機関からの要求に対する法的保全プロセスを定義する

**検証ゲート**:

- [ ] すべてのデータストアのデータ分類を実施している
- [ ] 所在地要件が確認され、インフラ設計にマッピングされている
- [ ] 保持ポリシーが文書化され、自動執行されている
- [ ] 削除権のプロセスがサービス開始前に設計されている

### 10. データドメインごとの単一の信頼できる情報源

**原則ステートメント**:
受注管理の各データドメインは、単一の権威あるシステム・オブ・レコードを持たなければならない。派生コピーは明示的に識別し、読み取り専用とし、定義されたスケジュールで同期する。

**根拠**:
移行期間中、レガシーシステムとモダンシステムが一時的にデータを共同管理する場合がある。明確なシステム・オブ・レコードがなければ、調整の失敗・データの乖離・注文状態の矛盾は避けられない。

**示唆**:

- 移行開始前に、各データドメイン（注文・顧客・商品・在庫）のシステム・オブ・レコードを特定する
- 並行稼働期間中は、一方のシステムが書き込み、他方は同期経由で読み取る
- 派生・キャッシュコピーはアーキテクチャドキュメントで明示的にラベル付けする
- 競合解決戦略のない双方向同期は禁止する
- 共有参照データ（商品カタログ・顧客識別情報など）についてマスターデータ管理戦略を定義する

**検証ゲート**:

- [ ] すべてのデータエンティティについてシステム・オブ・レコードが識別・文書化されている
- [ ] 並行稼働における同期方向が明示的に定義されている（双方向ではない）
- [ ] 競合解決戦略の文書なしに双方向同期が存在しない
- [ ] 派生コピーの同期遅延が計測され、定義されたSLAの範囲内にある

### 11. データ移行の完全性

**原則ステートメント**:
レガシーシステムから移行されたすべてのデータは、新システムで本番使用される前に定義された品質基準に対して検証しなければならない。移行は再現可能・監査可能・可逆的でなければならない。

**根拠**:
レガシーデータには、長年にわたって蓄積された不整合・非正規化・ビジネスルール違反が含まれていることが多い。未検証のデータを移行すると、新システムの初日からデータが汚染される。

**示唆**:

- 移行開始前にデータ品質ルールを定義・文書化する
- すべての移行実行に対して自動検証スイートを実行する
- 移行の失敗は処理を停止し、修正レポートを生成する。本番環境での部分的移行は禁止する
- 移行スクリプトはバージョン管理し、既知の状態から再実行可能にする
- 各実行後にレガシーのレコード件数と主要メトリクスを移行済みデータと照合した整合報告書を作成する

**検証ゲート**:

- [ ] データ品質ルールが文書化・自動化されており、最初の移行実行前に整備されている
- [ ] 各移行バッチ後に整合報告書が生成・レビューされている
- [ ] 移行スクリプトが冪等であり、再実行可能である
- [ ] 各移行フェーズのロールバック手順がテストされている

## III. アプリケーション原則

### 12. ドメイン駆動型モジュラーアーキテクチャ

**原則ステートメント**:
受注管理ドメインは、業務ケイパビリティに沿った境界コンテキスト (Bounded Context)（例: 注文受付・フルフィルメント・在庫・返品・通知など）に分解しなければならない。各境界コンテキストは自身のデータを所有し、公開インタフェースを通じて振る舞いを公開する。

**根拠**:
業務ケイパビリティに沿ったモジュラーアーキテクチャにより、独立したデプロイ・チームの自律性・段階的な移行が可能になる。また、アーキテクチャが技術的負債を蓄積するのではなく、業務とともに進化することを保証する。

**示唆**:

- 設計前に、ドメインエキスパートとの協働によって境界コンテキストを特定する
- 各境界コンテキストには、そのライフサイクルを担う単一チームを割り当てる
- 境界コンテキストをまたぐ直接的なデータベースアクセスを禁止する
- コンテキスト間のデータ交換は、公開されたイベントまたはAPIを通じて行う
- コンテキストマップによって境界コンテキスト間の関係を文書化する

**検証ゲート**:

- [ ] 境界コンテキストが明示的な境界と責務とともに文書化されている
- [ ] 関係性（上流/下流・共有カーネルなど）を示すコンテキストマップが作成されている
- [ ] コンテキストをまたぐデータベース依存関係が存在しない
- [ ] 境界コンテキストごとにチームオーナーシップが割り当てられている

---

### 13. API ファースト統合

**原則ステートメント**:
システムのすべてのケイパビリティは、利用者向けまたは連携用の機能を構築する前に、バージョン管理された文書化済みのインタフェースを通じて公開しなければならない。インタフェース仕様はプロデューサーとコンシューマー間の契約である。

**根拠**:
API ファーストにより、統合の契約が明示的・テスト可能・安定したものになる。内部の実装の詳細への密結合を防ぎ、プロデューサーとコンシューマーの並行開発を可能にする。

**示唆**:

- 実装開始前にインタフェース仕様（契約）を作成・レビューする
- すべてのインタフェースをバージョン管理し、後方互換性と非推奨化の戦略を文書化する
- 破壊的変更には新しいインタフェースバージョンが必要であり、暗黙の非互換性は禁止する
- システム境界をまたぐ直接的なデータベースアクセスを禁止し、すべての統合は公開インタフェースを通じて行う
- インタフェース仕様を内部の開発者カタログに公開する

**検証ゲート**:

- [ ] 実装開始前にインタフェース仕様が公開されている
- [ ] バージョニング戦略が定義・文書化されている
- [ ] すべての統合ポイントに対する契約テストがCIパイプラインに組み込まれている
- [ ] 破壊的変更の検出がCIパイプラインで自動化されている

---

### 14. イベント駆動型注文ライフサイクル

**原則ステートメント**:
注文ライフサイクルの状態遷移（作成・確定・フルフィルメント・出荷・配達・キャンセル・返金）はドメインイベントとして発行しなければならない。ダウンストリームシステムは、ポーリングや同期的な呼び出しではなく、非同期でイベントを消費する。

**根拠**:
イベント駆動型統合は、注文処理をダウンストリームシステム（通知・分析・ERP・物流）から分離する。レジリエンスの向上・独立したスケーリング・注文状態変更の自然な監査ストリームを実現する。

**非同期イベントが適用される場面**:

- 注文状態変更（注文ライフサイクルにおけるすべての遷移）
- 在庫引き当てと解放
- フルフィルメントと物流トリガー
- 顧客通知の送信
- 分析・レポートのデータフィード

**同期呼び出しが許容される場面**:

- 即時フィードバックを必要とするリアルタイムなユーザー操作（注文受付確認）
- 冪等な読み取りクエリ（注文ステータス照会）
- 即時結果を必要とする決済認可

**検証ゲート**:

- [ ] すべての注文ライフサイクル遷移に対してドメインイベントが定義されている
- [ ] イベントスキーマがバージョン管理されており、スキーマレジストリまたはカタログに公開されている
- [ ] すべてのイベントコンシューマーにデッドレターキューとエラー処理が設定されている
- [ ] イベントコンシューマーが冪等である（重複イベントを受信しても同じ結果を返す）
- [ ] 注文処理から非クリティカルなダウンストリームシステムへの同期呼び出しが存在しない

---

### 15. レガシーシステム腐敗防止

**原則ステートメント**:
レガシー受注管理システムとのすべての統合は、腐敗防止層 (Anti-Corruption Layer) を通じて仲介しなければならない。モダンドメインモデルは、レガシーのデータ構造・命名規則・ビジネスルール違反によって汚染されてはならない。

**根拠**:
レガシーシステムは数十年にわたる技術的負債とビジネスルールの逸脱を蓄積している。レガシーの構造への直接的な結合は、これらの問題を新システムに持ち込み、モダナイゼーションをより複雑かつ高コストにする。

**示唆**:

- 腐敗防止層はレガシーとモダンのデータモデル間の変換を行う
- モダンドメインモデルはレガシーの逆リエンジニアリングではなく、業務要件から設計する
- レガシー固有の概念（エラーコード・ステータスコード・フィールド命名）はモダンの境界コンテキストに漏洩させない
- 腐敗防止層は、計画された廃止予定日を持つ一時的なインフラである

**検証ゲート**:

- [ ] 腐敗防止層が変換マッピングとともに文書化されている
- [ ] モダンドメインモデルがレガシー構造から独立してレビューされている
- [ ] 腐敗防止層の廃止マイルストーンがプロジェクトロードマップに設定されている
- [ ] レガシー固有の用語がモダンのAPIおよびイベントスキーマに存在しない

## IV. 品質特性

### 16. パフォーマンスとスループット

**原則ステートメント**:
すべてのシステムは、計算リソースを効率的に使用しながら、想定される負荷のもとで定義されたパフォーマンス目標を達成しなければならない。パフォーマンス要件は、デプロイ後に計測するのではなく、設計前に定義する。

**受注管理のパフォーマンス目標**（コンポーネントごとに確認が必要）:

- **注文受付レスポンスタイム**: 通常負荷時 p95 < 3秒
- **注文ステータス照会レスポンスタイム**: p99 < 1秒
- **注文処理スループット**: ピーク注文量予測に2倍のヘッドルームを加えた値
- **バッチ移行スループット**: 各移行フェーズが定義されたメンテナンスウィンドウ内で完了できる処理速度

**示唆**:

- パフォーマンス要件は設計前に要件成果物に文書化する
- 各本番リリース前に、想定されるピーク容量でロードテストを実施する
- 読み取りの多い操作（商品カタログ・顧客プロフィールの参照）にキャッシュ戦略を定義する
- 即時応答を必要としない操作には非同期処理を採用する

**検証ゲート**:

- [ ] 計測可能な目標値を持つパフォーマンス要件が文書化されている
- [ ] 想定ピークの2倍の負荷でロードテストを実施している
- [ ] クリティカルパスのパフォーマンス回帰テストがCIパイプラインに組み込まれている
- [ ] 高頻度の読み取り操作のキャッシュ戦略が文書化されている

### 17. 可用性と障害復旧

**原則ステートメント**:
すべてのシステムは、自動回復と最小限のデータ損失を伴う定義された可用性目標を達成しなければならない。注文データはミッションクリティカルであり、最後に確認されたトランザクションの時点まで回復可能でなければならない。

**可用性目標**（コンポーネントレベルごとに確認が必要）:

- **注文受付・管理**: 月間可用性 99.9%（月間最大43.8分の計画外ダウンタイム）
- **注文ステータス参照パス**: 月間可用性 99.9%
- **管理・レポートシステム**: 月間可用性 99.5%
- **目標復旧時間 (RTO)**: レベルごとに定義。注文処理レベルは最大4時間
- **目標復旧時点 (RPO)**: 注文データは最大15分

**示唆**:

- すべての本番コンポーネントで複数のアベイラビリティゾーンにまたがる冗長性を確保する
- すべてのステートフルコンポーネントに自動ヘルスチェックとフェイルオーバーを実装する
- 定期的な障害復旧テストを実施する（最低四半期ごと）
- バックアップと復元の手順を文書化・検証する

**検証ゲート**:

- [ ] コンポーネントレベルごとに可用性SLAが定義されている
- [ ] RTOとRPO要件が文書化されており、DR テストを通じて検証されている
- [ ] 本番ワークロードにマルチゾーン冗長性が実装されている
- [ ] バックアップと復元が、時間を計測した回復演習によって検証されている

### 18. 保守性と進化性

**原則ステートメント**:
すべてのシステムは変更を考慮して設計しなければならない。明確な責務の分離・モジュラーな境界・重要な選択を文書化するアーキテクチャ決定記録 (ADR) を備えること。

**根拠**:
受注管理システムは、ビジネスルール・価格モデル・商品種別・連携パートナーの進化に合わせて変化しなければならない。短期的なデリバリーを優先して保守性を犠牲にする設計上の意思決定は、複利的な技術的負債を生む。

**示唆**:

- 責務を文書化した明確なモジュール境界を定義する
- ビジネスロジック・データアクセス・統合の懸念事項を分離する
- すべての重要なアーキテクチャ上の選択についてADRを作成する
- 自信を持ってリファクタリングできる十分な自動テストカバレッジを確保する
- レガシーコンポーネントへの依存を追跡し、継続的に削減する

**検証ゲート**:

- [ ] アーキテクチャドキュメントが最新であり、各フェーズゲートでレビューされている
- [ ] すべての重要な設計上の選択についてADRが作成されている
- [ ] クリティカルなビジネスロジックについて定義された閾値を満たす自動テストカバレッジがある
- [ ] 各設計レビューでモジュール依存関係グラフがレビューされている

## V. 開発プラクティス

### 19. Infrastructure as Code (IaC)

**原則ステートメント**:
すべてのインフラはバージョン管理されたコードとして定義し、自動化されたパイプラインを通じてデプロイしなければならない。本番インフラへの手動変更は禁止する。

**根拠**:
手動によるインフラ変更は、文書化されていない状態・環境間の不整合・インシデント後の回復の遅延を生む。Infrastructure as Code (IaC) により、環境の再現性と変更の監査可能性を確保する。

**示唆**:

- すべてのインフラを宣言的なバージョン管理されたコードで定義する
- インフラ変更はデプロイ前にコードレビューを受ける
- 開発・ステージング・本番の各環境をコードから再現可能にする
- 本番への手動変更を禁止し、すべての変更をパイプライン経由で実施する
- インフラコードをアプリケーションコードと並行してバージョン管理する

**検証ゲート**:

- [ ] すべてのインフラがバージョン管理されたコードとして定義されている
- [ ] インフラ変更の自動デプロイパイプラインが整備されている
- [ ] ステージングと本番の環境同一性が確保されている
- [ ] 未追跡の手動変更がないことを確認するインフラドリフト検出が実施されている

---

### 20. 自動化テスト

**原則ステートメント**:
すべてのコード変更は、開発環境より上位の環境へのデプロイ前に自動テストによって検証しなければならない。テストカバレッジはクリティカルな注文処理パスを含まなければならない。

**テストピラミッド**:

- **ユニットテスト**: 高速・独立・ビジネスロジックの高カバレッジ（ビジネスロジックで80%以上を目標）
- **統合テスト**: コンポーネント間の相互作用・データベース操作・メッセージング（主要な統合パスのカバレッジを目標）
- **契約テスト**: 境界コンテキスト間および外部連携のインタフェース契約
- **エンドツーエンドテスト**: 受注から出荷確認までのクリティカルな注文フロー
- **パフォーマンステスト**: ピーク負荷時の注文スループットとレイテンシー

**検証ゲート**:

- [ ] 自動テストがマージとデプロイのゲートとなっている
- [ ] 注文ドメインロジックのユニットテストカバレッジが定義された閾値を満たしている
- [ ] すべての統合ポイントに対する契約テストがパイプラインに組み込まれている
- [ ] エンドツーエンドテストが最低限、注文受付・決済失敗・キャンセル・返金フローをカバーしている
- [ ] パフォーマンステストが各主要リリース前にプリプロダクション環境で実行されている

---

### 21. 継続的インテグレーションとデプロイメント

**原則ステートメント**:
すべてのコード変更は、各ステージで定義された品質ゲートを持つ、自動化されたビルド・テスト・セキュリティスキャン・デプロイのパイプラインを通過しなければならない。

**パイプラインステージ**:

1. **ソース管理**: すべての変更をピアレビューとともにバージョン管理にコミットする
2. **ビルド**: 自動コンパイル・パッケージング・依存関係の解決
3. **テスト**: 自動ユニットテスト・統合テスト・契約テストの実行
4. **セキュリティスキャン**: 依存関係の脆弱性スキャンと静的コード解析
5. **デプロイ**: デプロイ後のスモークテストを伴う自動デプロイ
6. **本番ゲート**: 移行フェーズ中の本番デプロイには手動承認ゲートを設ける

**品質ゲート**:

- 次の環境へ昇格する前にすべてのテストが通過していなければならない
- 未解決の重大・高度なセキュリティ脆弱性がないこと
- 2人目のエンジニアによるコードレビュー承認が必要である
- 本番デプロイには文書化されたロールバック計画が必要である

**検証ゲート**:

- [ ] すべてのサービスに自動CI/CDパイプラインが整備されている
- [ ] セキュリティスキャンがパイプラインに統合されている
- [ ] デプロイが既知のアーティファクトから自動・再現可能に実施される
- [ ] 各サービスのロールバック機能がテストされている
- [ ] 本番デプロイの承認プロセスが文書化されている

## VI. 適用除外プロセス

### アーキテクチャ適用除外の申請

本原則は、アーキテクチャレビュー委員会 (ARB) が承認した文書化された適用除外がある場合を除き、遵守が必須である。

**適用除外が認められる正当な理由**:

- 準拠することが実質的に非実用的となる技術的制約
- 原則と相反する法的・規制的要件
- 段階的移行フェーズにおける過渡的な状態（終了日が必要）
- 本番トラフィックのない時間限定のPoC（概念実証）またはスパイク

**適用除外申請に必要な情報**:
